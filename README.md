# 秋招助手 · Autumn PM Coach

> 面试后的 PM 诊断与成长记忆工作台：把一场面试整理成**可追溯诊断总结**，把多场面试沉淀成**短期训练动作与长期能力趋势**。不做实时模拟面试。

- **在线只读演示**：<https://huggingface.co/spaces/Vergessne/autumn-pm-coach>
- **纯 Python 标准库**，无第三方依赖，本地一条命令即可运行
- **数据默认留在本机**，复盘 / 搜索仅在你主动发起时才调用大模型

### 30 秒看懂

| 输入 | 诊断总结 | 短期记忆 | 长期记忆 |
| --- | --- | --- | --- |
| JD、简历、面后速记 / 转写 | 原文证据、六项 PM 能力、评分理由、缺口和置信度 | 本场最高优先级、下一条追问、训练题和验收标准 | 跨场重复缺口、能力趋势、未完成行动和证据时间线 |

搜索 Agent 只负责为 JD 和问题生成提供公开上下文：规划查询、筛选候选、尝试读取公开页面、去重并在预算耗尽时停止。确定性代码负责相关度、来源状态和证据门槛。

---

## 它解决什么问题

面试最大的浪费，是「面完就忘」：

- 面试当下紧张，记不清自己到底哪句答得虚、哪个追问没接住。
- 复盘全靠感觉，没有原文证据，容易自我安慰或过度否定。
- 面了十几场，却说不清自己在「指标定义」「项目主导力」这些能力上到底有没有进步。

这个工具把这条链路结构化，让每一场面试都能沉淀成可复算的成长信号。

## 整体思路

产品由三个核心能力串成一条闭环，另有一组把原始材料变成结构化文本的输入处理支撑它们：

```text
  面后手写记录 / 速记问卷 / 录音转写（输入处理）
        │
        ▼
  单场复盘 ───────────► 每条结论都带原文证据 + 六维能力评分 + 行动项
        │  落盘到本地记忆
        ▼
  确定性长期记忆 ─────► 统计重复薄弱点 / 能力均分变化 / 未完成行动项
        │  多场累积
        ▼
  阶段成长报告 ───────► 跨场趋势解读（须有证据，显式标注数据是否足够）

  JD 研究 Agent ──（搜索 + 公开页读取 + 预审分级）──► 只有「人工确认」的资料才进复盘引用
```

设计上贯穿一条原则：**理解和归纳可以交给模型，但可信度由确定性代码来守。**
搜索结果不当事实、证据必须能在原文里定位、跨场趋势用可复算的统计得出——
这样即使模型出错，系统也不会把错误当成事实。

## 核心能力

### 一、JD 驱动的研究 Agent

粘贴 JD 并解析后，岗位关键词、面试重点、搜索主题和同义词会直接进入研究 Agent；生成面后速记问题时，也会把这些研究线索作为“参考上下文”，而不是另起一套手工搜索。

Agent 的固定闭环是：

1. 规划公司 / 岗位 / 轮次 / 主题的查询变体；
2. 通过公开搜索发现候选 URL，并尝试读取限定平台的公开 HTML；
3. 对限定平台的公开 HTML 做一次短时、无登录、无 Cookie 的读取，自动回填标题、规范链接和可见摘要；
4. 去重、计算相关度、标记页面状态，在达到数量或预算后停止。

资料库里的“仅搜索候选预览”按钮只返回带搜索来源的候选链接，并对限定平台尝试一次有边界的公开 HTML 读取；JD 解析后的自动流程和“Agent 自动调研并收录”会逐条去重、初筛、自动回填标题/链接/可见正文并持久化候选。自动读取不等于人工确认，两条路径都不会把搜索摘要直接当成原帖事实。

小红书的“公开路由”指 `site:xiaohongshu.com/explore` / `xhslink.com` 的公开网页搜索和页面读取，不是登录态 API、Cookie 抓取或私密内容访问。公开页面可能只返回脚本壳、拒绝访问或没有正文；这时系统会展示原因并保留 `待人工确认`，不会把搜索摘要当成原帖事实。

网上的面经真假难辨、时效性差、可能是营销，直接采信就是在放大不可信信息。因此外部资料仍有一道**证据门槛**：

1. **搜索只给线索**：发现候选帖子的 URL / 标题 / 摘要，明确标注「这是发现，不是已验证证据」；返回的链接必须来自搜索引用，不允许编造 URL。
2. **自动读取不等于可信**：后端会尝试自动打开公开原帖，自动回填标题、规范链接和可见正文，并触发一次 AI 预审，减少逐条复制成本；结果仍标为 `auto_fetched_unverified`，不会自动确认原帖真实性。脚本壳、拒绝访问或正文不足时才需要人工打开并补充摘录。
3. **三级分级**：`AI 预审通过 · 待确认`（置信度 ≥80 且有具体细节、无矛盾）/ `待人工确认` / `不使用`。AI 预审通过仍只能生成问题线索，只有用户确认后才进入事实型复盘引用；置信度 <80 会被强制降级为待人工确认。
4. **评论只作弱信号**：匿名评论只能微调置信度，永远不能单独当作事实证明。
5. **可溯源 + 精准引用**：每条资料保留原链接与发布日期；生成复盘时优先引用与当前公司 / 岗位匹配的可用资料，泛用资料仅作兜底。

### 二、单场面试表现的评价

面试复盘最容易变成「泛泛而谈」。这里用**证据优先**把评价钉在事实上：

1. **输入当不可信数据**：面试记录、JD 在处理时被显式声明为「待分析材料」，避免记录里夹带的指令劫持模型。
2. **固定结构产出**：总结 / 表现亮点 / 需要补强 / 六维 PM 能力诊断 / 逐题复盘 / 下一步行动项。
3. **证据逐条校验**：每条结论都要引用你记录里的原文，后端会把证据拿去和原文做匹配，**定位不到的引用直接替换成「（无原文可佐证）」**，绝不让判断性描述冒充原文。评分统一钳制到 1–5。

六维 PM 能力维度：

```text
产品判断 · 项目主导力 · 指标与实验 · 推进与协作 · 结构化表达 · 业务与岗位理解
```

### 本场短期记忆

每次复盘会留下一个可执行的“训练快照”：本场训练分、证据覆盖度、最高优先级能力、具体缺口、下一条追问、训练题和验收标准。它回答的是“下一场面试前先练什么”，不会把一场样本直接外推成候选人的长期能力。

### 三、跨多场面试的长期总结

单场复盘只看到「这一次」，真正的成长信号藏在「多场之间的重复」里。

1. **确定性长期记忆**：从所有**已复盘**的面试里，用统计（不是模型）聚合出——重复出现的薄弱点计数、六维能力的平均分（按最弱排序）与最新分、未完成的行动项、面试时间线。
2. **阶段成长报告**：在这份可复算的记忆之上生成跨场趋势解读；重复缺口、次数、来源和训练行动由后端确定性投影，`report_grounding` 记录输入边界，模型只补充匹配对象的解释，并显式输出「哪些数据太少、不足以下结论」。
3. **不编趋势**：数字和计数来自确定性统计，模型只负责解释，避免把偶然当规律。

## 输入处理（记录 / 简历 / JD）

这几项把原始材料变成结构化文本，服务于上面三个核心能力：

- **面后手写记录 / 速记问卷（推荐）**：直接手写这场面试的问答与感受即可；也可用「面后速记问卷」按 JD × 简历预判考点、逐条作答后一键并入记录，**全程无需录音**。
- **录音转写（备选，需合规确认）**：如已取得对方同意，可上传录音转成文字（部分公司 / 地区禁止面试录音，请先确认合规）。也支持直接导入 .txt / .md / .srt。
- **简历解析**：上传 PDF / PNG / JPG / WebP，解析成可编辑文本，维护多个简历版本。
- **JD 提取**：粘贴岗位 JD，一键提取岗位画像（职责 / 要求 / 关键词 / 可能追问）。

> 联网资料发现与音频 / 文件解析走 Gemini 原生 API（Google Search / Files），文本备用模型无法完全替代这两项。

## 🚀 快速开始

需要 **Python 3.12+**（仅标准库，无需 `pip install`）。

```bash
# 1. 克隆仓库
git clone https://github.com/Vergessen428/project_resume.git
cd project_resume

# 2. 配置模型 Key（至少填一个供应商）
cp app/.env.example app/.env
#   编辑 app/.env，填入 GEMINI_API_KEY 或 OPENAI_API_KEY 等
```

## 🧭 使用流程

```text
① 新建面试        →  填公司 / 岗位 / 轮次，粘贴 JD（解析后自动触发研究 Agent）
② 记录面试内容    →  手写这场面试的问答与感受，或用速记问卷逐条作答并一键并入（备选：上传录音转写）
③ 生成 PM 复盘    →  产出「表现亮点 / 需要补强 / 能力诊断 / 逐题复盘 / 行动项」，每条都带原文证据
④ 关联研究上下文  →  Agent 自动规划查询、尝试读取公开原帖并标注来源状态，问题生成引用候选线索
⑤ 阶段成长报告    →  跨多场面试提取重复薄弱点和能力变化
⑥ 回填面后结果    →  在面试档案的「面试结果（自报，可选）」选择等待结果 / 通过 / 未通过并保存
```

### 结果反馈怎么用

结果反馈入口就在「面试复盘」页顶部的**面试档案**区，不需要重新生成本场复盘。面试结束后先完成复盘；收到结果后再回来选择「通过」或「未通过」，等待通知时可先选「等待结果」。保存后，字段会以 `outcome_source=self_reported` 写入长期记忆，用于统计不同场次的训练反馈。

它不会改写本场 1–5 分、训练分或证据，也不代表录用概率。少于 4 场只展示样本不足；达到 4 场才做描述性分组；只有同一评分口径下通过和未通过各至少 2 场，才展示较稳定的方向性观察。静态 Demo 已展示入口和状态，但所有写入仍是只读提示。

## 🔀 多模型兜底

文本复盘、JD 提取、资料预审和阶段报告支持按顺序自动切换模型：

```text
Gemini → OpenAI → DeepSeek → OpenRouter → Custom (任意 OpenAI 兼容端点)
```

在 `app/.env` 填入至少两个供应商的 Key，按需修改 `MODEL_FALLBACK_ORDER`。未配置的供应商

## 🧱 技术栈与架构

- **后端**：Python 3.12+ 标准库 `http.server`，零第三方依赖
- **前端**：原生 HTML / CSS / JS，无构建步骤
- **模型**：Gemini / OpenAI / DeepSeek / OpenRouter / 自定义 OpenAI 兼容端点
- **存储**：本地 JSON（原子写入 + 进程内线程锁），默认 `.gitignore`；不承诺多进程/多实例并发安全
- **测试 / CI**：`unittest` + GitHub Actions（push / PR 自动跑）

```text
app/
  web_app.py          # HTTP 服务与路由、三级访问控制、限流
  core/
    model_provider.py # 从环境变量装配模型客户端（唯一 provider 装配点）
    models.py         # OpenAI 兼容 / Gemini / 多模型兜底，统一 complete(prompt)
    multipart.py      # 标准库 multipart 解析（替代已弃用的 cgi）
    interview_review.py      # 单场复盘：组 prompt、证据校验、评分钳制
    research_grounding.py    # 面经搜索、可用性预审、有边界的调研 agent
    audio_transcription.py   # 录音 / 简历文件解析（备选输入）
    growth_memory.py         # 确定性长期记忆聚合（统计，非模型）
    pm_skills.py             # 六维 PM 能力定义与评分锚点
    interview_store.py / resume_store.py / research_store.py  # 本地 JSON 存储
  web/                # 前端（原生 HTML/CSS/JS）
  data/               # 本地 JSON 数据（gitignore）
static_demo/          # 纯前端只读演示版（部署到 Hugging Face Static Space）
tests/                # 单元测试
docs/                 # 项目文档与界面截图
```

## 🔐 数据与隐私

- 面试记录、简历、面经默认保存在本机 `app/data/`（`interviews.json` 等），已被 Git 忽略。
- **本地保存不等于不会出网**：用户主动点击复盘、JD 搜索、速记出题或录音/文件解析时，必要的 JD、简历、转写、个人笔记或搜索词会发送给当前配置的模型/搜索服务；其留存和训练政策由服务商决定。
- 静态 Demo 不发送真实资料。公网后端必须设置 `APP_ACCESS_TOKEN`，真实资料不应直接放入公开演示实例。
- 完整版复盘提供可选的“发送前尽力替换公司名”开关；它只替换用户填写的公司名，不是完整匿名化，也不会覆盖搜索服务或其他敏感信息。
- 动态版和静态版会对手机号、邮箱、身份证号做确定性的“可能包含敏感信息提示”。它只提醒用户发送前自行删减，不阻止操作、不自动上传、不代表完整脱敏，也不能替代服务商隐私政策核对。
- 本地完整版提供记录删除和 `/api/export` 数据导出；删除面试后，长期记忆会从剩余记录重新计算。
- 三级访问控制：本地免密 / 公网需 `APP_ACCESS_TOKEN` / `APP_DEMO_MODE=1` 只读演示。
- 口令连错 5 次锁定 60 秒（按 IP 限流），防暴力破解。
- 上传的音频和简历文件只写入受控的本地临时目录，处理完成后删除；应用启动时还会清理上次异常中断遗留的本程序临时文件，不把音频写入长期数据 store。

### PM Coach V2

训练行动项会保留来源面试 ID、日期、公司和轮次；长期成长报告会显示每条训练建议来自哪场面试，便于回溯“诊断 → 训练 → 下一场验证”，而不是把跨场建议当成无来源的泛化结论。

复盘 schema 当前为 `2.1`：始终输出六项 PM 能力，每项拆成四个有权重的可观察子维度。后端校验原文证据、按固定权重计算确定性的 `exact_score`，再映射到 1–5 分；未覆盖的能力会显式显示为“未评分”，不会被模型随手给出的总分填充。输出还包括训练分、证据覆盖度、置信度、能力缺口、下一条追问和训练验收标准。每次复盘还记录实际 `scored_by`（provider、model、prompt/rubric 版本和时间），方便识别跨模型趋势。训练分只表示练习优先级，不代表录用概率或候选人价值。

面向中文 PM 用户的 Skill 规则已集中放在 [`skills/pm-interview-evaluation/`](./skills/pm-interview-evaluation/)：主流程、评分规则、证据政策、诊断 Schema、追问训练和小红书搜索边界均为中文说明。为了保持代码和历史数据兼容，文档中的 `skill_id`、字段名和状态值仍保留英文原值；中文文本负责解释含义和使用边界。

面试记录支持可选的自报结果 `passed / failed / pending`，只有明确写入 `outcome_source=self_reported` 的记录才会进入结果反馈；旧数据缺少来源字段时仍可读取，但不会被默认为自报。长期记忆在 4 场结果后只展示描述性分组，6 场且通过/未通过各至少 2 场后才展示较稳定的方向性观察；混合模型、历史未知模型或样本不足时会明确降级。outcome 不用于推断录用概率，也不证明分数导致了结果。

面经资料库支持 `全网 | 小红书 | 牛客`。解析 JD 后会自动生成搜索主题和同义词，交给 Agent 规划查询、去重候选、尝试打开具体公开原帖，并自动把候选写入资料库；平台首页、泛入口和非白名单链接不会作为原帖候选。相同查询计划在默认 60 分钟内已有足够候选时会复用本地研究缓存，不重复消耗联网搜索配额；缓存只复用候选，不改变来源状态。拿到足够公开正文时还会自动触发 AI 预审，不要求用户逐条复制。小红书的“公开路由”指 `site:xiaohongshu.com/explore` / `xhslink.com` 这类公开网页搜索和页面读取，不是登录态 API 或 Cookie 抓取；自动读取结果仍带 `auto_fetched_unverified`，AI 预审通过也只标为 `auto_approved` 问题线索，只有用户确认后的 `approved` 才能进入事实型复盘引用。脚本壳或访问受限时才提示补充原帖摘录。

面后速记问题会同时读取 JD 结构化拆解、自动收录的候选资料和已确认资料。点击“生成问题”时，如果用户还没先解析 JD，前端会幂等地先补一次 JD 拆解和公开搜索；搜索失败会明确降级为通用问题，不伪装成已参考资料。研究 Agent 会从可读取正文中提取有限的出题线索，并在无法核对原文时标为 `unverified`；候选资料只影响“应该追问什么”，不会改变“面试中实际发生了什么”；最终复盘仍只用转写原文作为表现证据。面试保存的研究快照带有来源 ID，资料被删除或降级后会自动失去引用资格，但仍可作为历史线索查看。

研究上下文的“已确认”状态必须绑定到本地资料库中的 `research_id`；客户端直接提交的 `status=approved` 快照不会绕过这个绑定校验，只能作为待核对的出题线索。

如果同一公开链接之前只读到脚本壳，后续搜索再次获得足够正文时，后端会自动补齐空候选并重新预审；已有人工摘录不会被自动内容覆盖。

研究 Agent、面试复盘和音频转写支持转为本地后台任务：接口立即返回 `202 + task.id`，前端轮询 `/api/tasks/{id}` 显示排队、执行、成功、失败、超时、取消或重启中断，并在仍有额度时允许按原步骤重试。`POST /api/tasks/{id}/cancel` 可以取消排队任务；运行中的网络调用采用协作式取消，晚到结果会被丢弃。研究/复盘 JSON 请求传 `async=true`，音频上传附带同名 multipart 字段。`tasks.json` 只保存任务类型、状态、尝试次数和安全错误，不持久化转写、JD、任务输入、结果或 runner；重启时未完成任务会标记为 `abandoned`，不能自动恢复，因此它不是分布式队列或生产级异步基础设施。

后台任务生命周期会写入受限的 `operations.jsonl`，只允许任务 ID、类型、状态、尝试次数、路由、错误类别、provider/model 和耗时等运维字段；日志会按大小轮换，不记录 prompt、模型原始响应、JD、转写、面经正文、邮箱、手机号或原帖 URL。`GET /api/operations` 只返回这组安全事件，供本地故障排查使用。

静态 Demo 使用 `demo_data.js` 注入平台相关的合成候选与 Agent 轨迹，页面展示完整六项诊断和一张明确标注为合成演示的小红书候选卡。查询、JD 解析、Agent 和成长报告按钮只展示带 `demo_only` 的预置轨迹，不联网、不调用模型、不写入资料库；其他写操作仍显示只读提示。

训练行动项支持 `pending / passed / needs_retry` 验收状态、完成时间和备注，并可以按“前测 → 重写 → 后测”记录训练回答；行动具有稳定 ID，重新生成复盘不会丢失已有尝试、通过状态或完成时间。后端强制阶段顺序，后测不存在时不会把行动伪装成完整闭环。长期记忆支持展开来源、重命名或忽略缺口，并提供确定性聚合的 `memory_version`、生成时间和输入审计；缺口治理的修改、忽略、删除会额外记录在不含转写正文的事件历史中，成长页和 `/api/memory/overrides/audit` 可查看；带确认调用 `POST /api/memory/overrides/audit/{event_id}/revert` 可恢复某次操作之前的标签状态，撤销本身也会留痕。`/api/growth-memory/replay` 可回看聚合边界。完整版成长页还提供带 SHA-256 完整性校验的版本化本地备份、恢复、导出；每次备份默认只轮换最近 7 份已校验副本，不会自动删除损坏或旧版未校验副本。恢复会先验证备份，再以失败回滚为边界替换本地 store；如果回滚也失败，会写入 `recovery-required.json`，启动检查和健康接口会明确要求恢复。清除原始转写或面经正文后，仍可编辑这场面试的 outcome、备注和其他元数据；清理不会锁死结构化记录。另有仅清除转写或面经正文、保留结构化结果的选择性清理，以及显式确认的保留策略预览与执行接口，默认不自动删除；静态版只展示这些入口，不读写真实本地数据。

固定评测样例覆盖稀疏转写、伪造引用、非法 ID、矛盾指标、旧 schema，以及研究 Agent 的空结果、脚本壳、自动读取正文但未验证、重定向越界和网页诱导；`app/core/evaluation_harness.py` 还提供 `evaluate_quality_gold_fixtures()` 和 `evaluate_relevance_fixtures()`：前者用版本化人工标注约束检查分数范围、缺口一致性、证据数量、多轮追问的量化证据、训练行动可执行性和模型间分数漂移，后者回放人工相关等级并计算 Precision@K、Recall@K、NDCG。所有回放都不联网、不调用模型；质量与相关性 fixture 目前只是小规模校准起点，不是招聘真值，也不等同于搜索或模型质量已经完成校准。

`render.yaml` 和 [DEPLOY_RENDER.md](./DEPLOY_RENDER.md) 保留为可选的带后端方案；如果当前
Render 无法创建或构建，忽略它即可，不影响静态展示。后端公网部署时仍**务必设置
`APP_ACCESS_TOKEN`**；演示模式没有管理口令时现在会强制只读。`/api/*` 默认按来源 IP
限制为每分钟 120 次，可用 `APP_RATE_LIMIT_PER_MINUTE` 调整；这只是单实例限流，不等于多用户
隔离或完整的生产网关防护。

```bash
python3 -m unittest discover -s tests -v

python3 -B scripts/resilience_drill.py
```

GitHub Actions 会重复执行单测、离线评测、编译检查、静态资源 HTTP smoke 和本地恢复/任务故障演练，并在独立的
`browser-smoke` job 中运行固定版本的 Chromium 交互验收。HTTP smoke 只证明
静态托管入口和资源路径可加载，不替代浏览器端的移动端布局、点击路径和无障碍验收。Resilience
drill 使用临时目录调用生产恢复和任务代码，不触碰本机真实数据，也不联网。

本机浏览器的桌面/移动端运行态路径记录见
[docs/design/RUNTIME_ACCEPTANCE.md](./docs/design/RUNTIME_ACCEPTANCE.md)。其中明确区分静态 Demo 的预置 Agent 轨迹与动态后端的真实联网任务，避免把演示响应误当成已完成的搜索部署。

长期迭代的里程碑、验收条件和下一阶段风险见 [docs/ROADMAP_LONG_TERM.md](./docs/ROADMAP_LONG_TERM.md)。当前项目已完成可信诊断与 JD 研究链路的单机基础闭环，但还不是多用户 SaaS，也不应把 `outcome` 或训练分解释成录用预测。
