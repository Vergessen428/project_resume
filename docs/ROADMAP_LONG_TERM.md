# PM Coach 长期迭代目标

这份路线图用于 Goal 模式的持续迭代。目标不是把页面堆成一个“万能 AI 面试官”，而是把
「诊断总结 → 短期训练 → 长期记忆 → 结果反馈」做成可验证、可删除、可回放的个人工作流。

## 当前目标

让一次真实的面试复盘能够回答四件事：

1. 哪些判断由面试原文支持，哪些只是缺失或矛盾证据。
2. 为什么当前能力分数落在这个锚点，而不是只给一个模型印象分。
3. 下一次应该练什么，什么条件满足后才算练会。
4. JD 和公开资料怎样帮助准备问题，但不会污染对本场表现的事实判断。

## 里程碑

### G0 可信诊断契约（当前已落地）

- 六项 PM 能力，每项四个有权重子维度。
- `score`、`exact_score`、`evidence`、`rationale`、`confidence` 和 `scored_by` 同时保存。
- 研究快照只有绑定 `research_id` 并通过当前资料库门槛后，才允许作为 `citation_allowed` 的确认上下文；无 ID 的客户端快照只能保留为问题线索。
- 原文不足、证据矛盾、非法 ID、旧 schema 都有确定性降级规则。
- 训练分只表示练习优先级，不输出录用概率。

验收：固定 fixture 能证明不凭空补结果、不接受非法技能 ID、不把低置信结果显示成高置信。

### G1 JD 驱动研究链路（当前已落地，需持续校准）

- JD 解析产出岗位画像、关键词、搜索主题和同义词。
- Agent 在查询预算内自主选择搜索路径；后端自动去重、读取白名单公开 HTML、回填标题/规范链接/正文状态，并记录读取状态计数和失败原因；AI 预审还会提取有限的 `question_leads`，供面后出题引用。
- 候选由后端持久化，浏览器中断不会丢失；正文足够时触发 AI 预审。
- 同一 URL 先前只有脚本壳时，后续获得足够公开正文会自动补齐并重新进入预审；已有人工摘录不会被自动读取结果覆盖。
- 普通公开搜索接口和 JD/Agent 流程都会尝试白名单公开页读取，并返回 `public_discovery_with_bounded_fetch` 元数据；这仍不等于人工确认原帖，脚本壳、访问受限或正文不足时保留待确认状态。
- 相关性和可信度分开：相关性由固定六维权重计算，可信度必须经过正文门槛和人工/AI 状态门。
- 平台白名单与原帖路径分开校验：平台首页、泛入口和非具体帖子 URL 不进入候选或可用资料；公开页重定向在每一跳跟随前校验目标域名与平台，不允许跨平台或越界跳转。
- 小红书只走公开搜索与公开页面读取，不登录、不读 Cookie、不访问私密接口。
- 相同查询计划在短时本地缓存窗口内可复用已收集候选，缓存不改变候选的可信度状态；`APP_RESEARCH_CACHE_SECONDS=0` 可关闭缓存。

验收：空结果有查询轨迹和失败原因；脚本壳、重定向越界、网页提示注入、用户信息/任意端口 URL 和非白名单域名不能进入可用资料。`research_cases.json` 已由 Agent 回放 harness 消费。

### G2 短期训练闭环（基础闭环已落地）

- 每场复盘生成一个训练快照：最高优先级缺口、下一条追问、训练题和验收标准。
- 行动项记录 `pending / passed / needs_retry`、备注、完成时间和下一次验证问题；没有 `post_test` 时 API 拒绝 `passed`。
- 行动项支持 `pre_test → rewrite → post_test` 训练记录，后端强制阶段顺序并保留有限次尝试历史。
- 行动项使用稳定身份并在复盘重跑时合并已有训练状态，避免模型重新生成诊断覆盖前测、重写、后测和通过结果；行动来源保留面试 ID、公司、岗位和轮次。
- 面后速记可引用 JD 和研究线索，但候选资料不进入面试事实证据；`question_leads` 必须保留原文摘录状态，未验证线索不能冒充事实。`auto_approved` 只代表 AI 预审完成，只有人工确认后的 `approved` 才能进入 `approved_context`。

下一步：为每类缺口增加真实匿名样例，比较答案结构变化，而不是只比较模型分数。

### G3 长期记忆与 outcome 边界（来源回放已落地）

- 长期记忆使用确定性统计聚合重复缺口、能力趋势、开放行动和证据时间线。
- 阶段成长报告的结构化 pattern、次数、来源和训练行动由长期记忆确定性投影，模型只提供匹配对象的解释；`report_grounding` 记录边界。
- 能力趋势优先使用可比复盘中的 `exact_score`，旧记录没有该字段时回退到历史 `score`，并在记忆 `algorithm_version` 中记录聚合规则。
- `outcome` 只有在记录明确声明 `outcome_source=self_reported` 时才作为用户自报训练反馈；至少达到样本门槛且模型可比较时才展示描述性方向，旧记录缺少来源字段时不参与聚合。
- 删除单场记录后，长期记忆从剩余记录重算；历史未知评分模型不伪装成可比数据。
- 记忆输出 `memory_version`、生成时间、能力来源和 `audit` 输入清单；`/api/growth-memory/replay` 提供不含转写正文的回放边界。
- 缺口标签的修改、忽略和删除会写入不含转写正文的治理事件历史；成长页、导出和备份可以回看这段历史，并可通过带确认的接口确定性撤销单次操作。

下一步：增加批量治理操作的更细粒度语义，以及能力趋势的可视化回放。

### G4 数据生命周期与恢复（基础能力已落地）

- 导出、带 SHA-256 摘要的版本化备份、恢复、选择性清除已存在。
- 恢复前校验备份标识、版本、结构和摘要；跨 store 写入失败会尝试回滚，并返回明确失败状态。
- 保留策略先预览再确认，默认不自动运行，只清理原始转写和面经正文。
- 结构化诊断、来源元数据和长期统计不随原文清除而消失。
- 清除原始转写后仍可编辑面试元数据、outcome 和治理字段；清理不会把记录锁死。
- 动态版和静态版在输入、文件导入、录音转写和面后速记回填后提供非阻塞的“可能包含敏感信息”提示；当前只覆盖有限的手机号、邮箱、身份证号模式，不承诺完整脱敏。
- 备份创建后默认轮换最近 7 份已校验副本；损坏、不可读和旧版未校验副本不会被自动删除。
- 恢复回滚未完成时会留下恢复标记，启动检查、`/healthz` 和 `/api/health` 会失败并给出恢复边界。

下一步：用匿名样例评估敏感信息提示的漏报边界，并增加真正的恢复演练；不要把模式提示升级成“自动匿名化”承诺。

### G5 可评测性与模型校准（质量校准基础已落地）

- 建立匿名、最小化的黄金样例：稀疏回答、量化缺失、矛盾指标、多轮追问、不同模型版本；当前已有 `quality_gold_cases.json` 的人工标注约束样例。
- 用人工标注比较证据覆盖度、锚点命中、缺口一致性、追问可执行性和引用污染率；搜索候选另用 `relevance_cases.json` 评估 Precision@K、Recall@K 和 NDCG@K。
- 记录 `scored_by`、prompt/rubric 版本和变更原因；模型升级必须跑回归，不直接覆盖历史结论。
- 固定 review/research/quality-gold/relevance fixture 已有统一 `evaluation_harness`，可回归检查 schema、ID、分数、证据边界、Agent 来源安全、训练题可执行性、模型分数漂移和搜索排序；标注集仍然很小，不能当作人工质量基准的充分替代。
- `python3 -B -m app.core.evaluation_harness` 可以一次运行四类离线回放，输出统一 `suite_version` 和各子套件结果；回放不联网、不调用模型。
- 复盘 API 也有契约回归：真实路由输出完整六项 `schema_version: 2.1`、实际 `scored_by` 和证据覆盖度，并验证缺失子维度不会被模型分数填充。

下一步：扩展匿名标注覆盖更多多轮追问、矛盾证据、量化缺失和不同 prompt/rubric 版本，并记录人工复核者、标注理由和变更历史。退出条件：连续两版模型在关键样例上没有新增伪证据，追问引用污染率受控，且评分漂移可解释。

### G6 真实部署硬化（未完成）

- 当前已增加单进程、按来源 IP 的 `/api/*` 请求限流和本地恢复失败健康检查；两者都不等于正式用户隔离或多实例网关。
- 非回环地址绑定在非 Demo 模式下没有 `APP_ACCESS_TOKEN` 会直接拒绝启动；这只是防止误暴露的 fail-closed 保护，不替代账号隔离。
- 研究 Agent、面试复盘和音频转写已经支持本地后台任务：前端可查询 `queued / running / cancel_requested / succeeded / failed / timed_out / cancelled / abandoned`，失败任务按固定上限重试；`POST /api/tasks/{id}/cancel` 支持排队取消和运行中协作式取消；`tasks.json` 只保存不含原始输入和结果的状态元数据，重启时未完成任务会标记为 `abandoned`，不替代分布式任务队列。
- 任务生命周期写入大小受限、字段白名单的 `operations.jsonl`；`/api/operations` 只返回安全运维事件，不写入 prompt、模型原始响应、JD、转写或来源正文。
- 公开后端必须具备正式用户隔离、密钥管理、请求限流、任务超时、日志脱敏和错误追踪。
- 研究 Agent、复盘和转写应逐步改为后台任务，前端显示任务状态和可重试步骤。
- 当用户数、记录量或并发达到阈值时，再按“JSON → SQLite → Postgres”迁移，保留 store 接口。

在 G6 完成前，Render 只适合作为受口令保护的个人后端实验，不应承载未隔离的真实多用户数据。

### 本轮审计结论（2026-07-14）

本轮补充了问题级证据状态 `missing / unverified / verified` 的确定性归一化，并加强静态 Demo 契约测试：六项能力的 24 个子维度、`scored_by`、`outcome_signal`、JD 搜索轨迹、相关度分解和来源状态都必须在演示数据中存在。当前 164 个 unittest、四类离线 fixture 回放、版本化的 6 条相关性标注、脚本语法检查、真实临时 HTTP 静态服务器 smoke 和四项本地恢复/任务故障演练通过；质量黄金样例已覆盖矛盾实验追问和 JD 业务适配缺失，训练行动重跑合并也有回归测试。研究上下文还会拒绝未绑定实体 ID 的伪造 `approved` 快照，并能升级先前只有脚本壳的空候选。AI 预审候选即使满足置信度门槛也只能作问题线索，不能绕过人工确认成为引用资料。

但 Goal 仍未完成。下一轮的退出条件不是“页面看起来更完整”，而是：

- 扩展真实脱敏 query-document 标注，并把搜索宏平均指标设为可追踪的版本化基线；
- 静态资源 HTTP smoke 已进入 GitHub Actions；独立的 `browser-smoke` job 已固定 Node 20、Playwright 1.52.0 和 Chromium，覆盖复盘、XHS 候选、Agent 轨迹、JD 到速记问题、只读写操作和移动端溢出，并上传截图；仍需取得首次绿色 CI 记录；
- 完成一次备份损坏、恢复失败、任务超时和取消的真实演练；标准脚本已加入 CI，仍需在后续版本继续增加磁盘空间不足和进程中断场景；
- 明确 Render 公网后端的账号隔离、持久化、密钥、任务和日志方案，或者继续把它限制为个人实验实例；
- 对自动读取的小红书正文保留“可用候选”与“真实性已确认”的双状态，不把 AI 预审文案当作原帖事实。
- 阶段报告的趋势、次数、来源和训练行动继续由确定性 memory 重建；混用模型时报告解释层也强制显示可比性警告。

## Goal 每轮工作协议

每次迭代都要留下：

- 一个可复现的用户路径或失败案例；
- 一个确定性规则或数据契约变化；
- 至少一个回归测试；
- README / 设计文档对“现在能做什么”和“仍不能保证什么”的同步说明。

禁止把以下内容当作完成标准：模型输出看起来更长、搜索结果数量更多、静态 Demo 更像真的，或单场分数更高。
