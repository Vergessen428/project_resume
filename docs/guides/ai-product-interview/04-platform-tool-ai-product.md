# AI 产品经理面试准备四：平台与工具型 AI 产品

## 适用岗位

这类岗位包括 AI 工作流平台、Prompt 平台、企业内部 AI 工具、Agent 搭建平台和模型应用平台。面试官更关注产品架构、角色分层、配置能力、权限、版本、评估和降低使用门槛。

## 一、基础知识定义

### 1. 平台型 AI 产品

平台型产品不是只服务一个终端用户，而是为多个角色提供可复用的能力和基础设施。用户可能包括业务人员、产品经理、算法、研发、运营和管理员。

### 2. Tool、Skill、Agent 的区别

Tool 通常是可被调用的具体能力，例如搜索、数据库查询或发消息；Skill 是围绕某类任务沉淀的能力说明和流程；Agent 是根据目标选择上下文、Skill 和 Tool 并完成任务的执行者。

### 3. Prompt 平台

Prompt 平台不只是文本编辑器，还应支持版本、变量、测试样例、权限、评估、发布和回滚。否则用户无法知道一次修改是否真的让模型效果变好。

### 4. 平台的可配置性与易用性

配置项越多，能力越强，但学习成本、出错概率和维护成本也越高。平台设计要为不同角色提供不同层级的入口，常见任务提供模板，复杂用户再开放高级配置。

### 5. 平台评估体系

平台需要支持样例集、版本对比、人工标注、模型指标、业务指标和线上反馈，帮助使用者判断一个 Prompt、Skill 或 Agent 是否值得发布。

## 二、典型面试问答

### 1. 平台型 AI 产品和普通 AI 应用有什么区别？

**回答框架：** 普通应用围绕一个终端任务优化，平台型产品要解决能力复用、角色协作、配置、权限、版本和评估。平台的核心用户不一定是最终消费者，因此需要同时考虑搭建者、使用者和管理员。

### 2. 如何设计一个 Agent 搭建平台的核心流程？

**回答框架：** 可以是定义目标、选择模型、配置 Prompt、接入 Skill/Tool、设置知识和 Memory、用样例测试、发布和观察效果。流程要支持从模板快速开始，也要允许专业用户逐层调试。

### 3. Tool、Skill 和 Agent 在产品中应该如何分层？

**回答框架：** Tool 是原子能力，Skill 是围绕任务的可复用方法，Agent 是动态选择和执行这些能力的主体。分层的价值是让能力可以复用、替换和评估，而不是把所有逻辑写进一个长 Prompt。

### 4. 如何设计平台的角色和权限？

**回答框架：** 区分创建、编辑、测试、发布、查看数据和管理密钥等权限，并按环境隔离开发、测试和生产。涉及用户数据、自动执行和外部工具时，权限要更严格，且保留操作审计。

### 5. 为什么平台需要版本管理？

**回答框架：** Prompt、模型、知识库、Tool 和评估集任一变化都可能改变结果。版本管理可以让团队比较效果、回滚问题版本并解释线上变化，否则 bad case 出现时无法定位责任。

### 6. 如何降低用户从“不会用”到“用起来”的成本？

**回答框架：** 提供任务模板、示例数据、默认配置、即时预览和分步引导，同时把复杂配置隐藏在高级模式。首个成功任务比功能数量更重要，用户需要尽快看到结果并知道如何继续优化。

### 7. 如何评价一个 Prompt 平台？

**回答框架：** 看用户是否能快速创建任务、迭代效果、定位 bad case 并安全发布；平台指标包括首次成功时间、版本迭代效率、评估覆盖、发布回滚和实际使用情况。不能只看创建了多少 Prompt。

### 8. 为什么平台必须提供评估能力？

**回答框架：** 没有评估，用户只能凭感觉判断 Prompt 或 Agent 是否变好，容易出现局部样例很好、整体效果变差。平台要支持固定样例回放、版本对比、人工评价和线上反馈。

### 9. 如何设计一个 AI 产品的评估集？

**回答框架：** 先从真实任务抽取正常、边界和失败样例，再按场景和风险分层，定义人工可判断的标准。评估集要版本化，并且持续从线上 bad case 回流，但未经确认的模型输出不能直接成为标准答案。

### 10. 如何处理模型、Prompt 和知识库之间的版本关系？

**回答框架：** 发布时记录完整依赖快照，至少包含模型版本、Prompt 版本、Skill/Tool、知识库版本和评估集版本。出现质量变化时可以复现和回滚，不能只显示一个模糊的“Agent 版本”。

### 11. 平台如何支持多模型？

**回答框架：** 先抽象统一输入输出和能力标签，再根据任务选择模型，兼顾效果、延迟、成本和隐私。平台要让用户理解模型差异，而不是只提供一个下拉框；高风险任务还需要固定评估门槛。

### 12. 如何设计 Tool Calling 的产品配置？

**回答框架：** Tool 需要有清晰描述、参数 schema、权限范围、超时、失败返回和示例。产品要允许配置哪些任务可以自动调用、哪些需要确认，以及调用结果如何展示给用户。

### 13. Memory 在平台型产品中怎么设计？

**回答框架：** 先区分短期上下文和长期记忆，再定义记忆来源、有效期、可见范围、删除和纠错。不是所有聊天内容都应该自动进入长期记忆，只有经过确认且对后续任务有价值的信息才适合保留。

### 14. 平台如何控制 AI 成本？

**回答框架：** 提供调用量、Token、模型、任务和团队维度的成本可见性，再通过缓存、摘要、模型分级、限额和预算告警控制。成本策略不能只由财务决定，还要结合任务价值和质量护栏。

### 15. 平台上线后如何发现问题？

**回答框架：** 结合线上质量、延迟、工具失败、用户修改、人工接管、投诉和业务结果观察。平台要能定位到具体 Agent、Prompt、模型、知识版本和输入场景，否则只能知道“效果下降了”，无法继续优化。

## 三、准备重点

- 能从多个角色解释平台需求；
- 能讲清 Tool、Skill、Agent、Memory 和 RAG 的产品分层；
- 能设计版本、权限、评估、发布和回滚；
- 能解释如何降低使用门槛并保证高级能力；
- 能把平台指标和下游业务结果连接起来。
