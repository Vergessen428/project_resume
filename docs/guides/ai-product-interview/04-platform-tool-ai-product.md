# AI 产品经理面试准备四：平台与工具型 AI 产品

## 这类岗位到底在考什么

平台与工具型 AI 产品包括 Agent 搭建平台、Prompt 平台、企业内部 AI 工作台、知识库与 RAG 平台、模型应用平台和 AI 工作流工具。它们不只服务一个终端任务，而是让不同角色配置、复用、评估和发布 AI 能力。

因此面试官不会只问“你会不会调用模型”，还会看你能否处理平台产品的复杂矛盾：

```text
能力要足够强，但不能难用
配置要足够自由，但不能不可控
组件要能复用，但不能抽象过度
发布要足够快，但必须可评估、可追踪、可回滚
```

平台产品经理要同时理解搭建者、业务使用者、算法研发、运维和管理员的任务。平台的成功不是创建了多少 Prompt，而是用户能否更快做出可用能力，并且团队能安全地复用和持续改进。

## 一、基础知识：面试中要讲到什么程度

### 1. 平台和应用的区别

应用围绕一个具体用户任务优化闭环；平台提供可复用的能力、规则和管理机制，服务多个应用和角色。平台不能只堆功能，还要解决统一标准、依赖关系、权限、版本、质量和使用门槛。

### 2. 平台的三类核心角色

通常至少有搭建者、使用者和管理员。搭建者关心配置和调试，使用者关心任务是否完成，管理员关心权限、成本、审计和稳定性。算法和研发还会关心数据、评估和发布。一个角色的便利不能以另一个角色无法维护为代价。

### 3. Tool、Skill、Agent 的产品分层

Tool 是有明确输入输出的原子能力，如搜索、查询数据库或发送消息；Skill 是围绕一类任务沉淀的知识、步骤和约束；Agent 是根据目标、上下文和中间结果选择并执行能力的主体。分层的价值在于复用、替换、评估和治理，而不是把所有逻辑塞进一段长 Prompt。

### 4. Prompt 和结构化输出

Prompt 应该包含目标、上下文、约束、示例和输出要求；稳定的业务流程还应使用 JSON Schema、字段校验和错误处理，避免下游依赖自由文本。Prompt 的质量不能只靠人工阅读，需要配套版本和评估样例。

### 5. 评估集和回归测试

评估集包括正常、边界、长尾和高风险样例，并且要有人工可判断的标准。每次模型、Prompt、Tool、知识库或策略变化，都应该对固定样例回放，防止局部优化导致其他场景退化。线上 bad case 经过确认后才能回流成标准样例。

### 6. 依赖快照和版本管理

一个 Agent 的行为由模型、Prompt、Skill、Tool、Memory、知识库、路由策略和评估集共同决定。发布时要记录完整依赖快照，而不是只给 Agent 一个版本号。这样才能解释线上变化、比较版本、灰度和回滚。

### 7. 权限、环境和审计

平台通常需要区分开发、测试、生产环境，并按角色限制创建、编辑、调用、发布、查看数据和管理密钥等权限。涉及企业知识、外部工具和自动执行时，还要记录谁在何时以什么版本做了什么操作。

### 8. 可观测性

可观测性不仅是请求成功率。平台要能看到任务、Agent、模型、Prompt、检索、Tool 调用、延迟、Token、错误、人工修改和最终结果之间的关联，才能定位质量下降、成本上升和链路失败。

### 9. Memory 和知识库治理

短期上下文服务当前任务，长期记忆服务跨任务复用；知识库是外部可维护资料。三者都要有来源、范围、有效期、权限、删除和纠错机制。不是所有对话都适合自动写入 Memory，也不是所有文档都应该被所有 Agent 检索。

### 10. 平台成本与复用价值

平台成本来自模型、检索、工具、存储、运维和人工评估。平台价值也不能只看调用量，需要看首次成功时间、能力复用率、发布效率、任务完成、下游业务效果和每个有效任务的成本。没有复用和质量的调用量可能只是浪费。

### 11. 抽象层和能力契约

平台抽象不是把所有差异抹平，而是定义稳定的输入、输出、权限、错误和版本契约。模型、Tool 或知识源可以替换，但上层任务不应因为底层小改动就全部重写。抽象过早会限制真实场景，抽象过晚则会产生重复配置，需要通过多个相似需求验证边界。

### 12. Schema 和兼容性

结构化 schema 规定字段类型、必填项、枚举、错误码和兼容策略。平台升级时要考虑旧 Agent 是否还能读取新输出，Tool 参数改变是否会导致线上失败。兼容性不是研发内部问题，产品需要定义迁移提示、版本锁定、回滚和废弃周期。

### 13. 配置依赖图

一个 Agent 的配置可能依赖模型、Prompt、Tool、Skill、知识库、Memory 和路由策略。平台要能展示依赖关系和影响范围，发布前发现缺失权限、过期知识或评估集不匹配。否则用户看到的只是一个“已发布”按钮，出了问题却无法知道改了什么。

### 14. 多租户隔离

企业平台不仅要隔离用户，还要隔离团队、项目、知识、日志、密钥和费用。共享模板可以公开，但生产数据和长期记忆不能默认跨项目可见。隔离边界要在产品权限、检索过滤、日志展示和导出能力上同时生效。

### 15. Tool 治理

Tool 需要生命周期、负责人、权限、版本、调用预算、超时、错误码和弃用策略。高风险 Tool 应默认关闭或要求审批，不能因为被注册到平台就对所有 Agent 可用。工具的描述也要面向模型和人分别可理解，减少误调用。

### 16. 评估即平台能力

平台不仅要提供“运行样例”的按钮，还要帮助用户管理样例、标注、版本、指标、差异和发布门槛。好的评估体验会把 bad case 变成可复现任务，让用户知道应该修改 Prompt、Tool、知识还是交互，而不是只显示一个分数。

### 17. 发布、灰度和回滚

发布需要包含依赖快照、评估结果、审批、灰度范围和回滚点。线上出现质量、延迟、成本或安全问题时，要能快速切换到稳定组合，并保留失败版本供复盘。回滚不是“恢复上一个 Prompt”，而是恢复整套可复现依赖。

### 18. 追踪和隐私的平衡

完整 Trace 有助于定位问题，但原始 Prompt、用户输入和工具结果可能含有敏感数据。平台要支持脱敏、字段级隐藏、采样、保留期限和分角色查看，让调试信息足够用但不过度暴露。日志设计本身也是 AI 产品的权限设计。

### 19. Memory 的生命周期

记忆需要经过候选、确认、有效、过期、纠正和删除等状态。平台要知道记忆来源、写入主体、适用范围、影响的 Agent 和删除后的传播范围。只提供一个“记住全部对话”的开关，会带来隐私、错误传播和难以解释的问题。

### 20. 平台采用和生态

平台上线后需要模板、文档、示例、支持和共享机制，否则每个团队会重复造轮子。生态治理还包括模板 owner、质量评级、废弃标记和使用反馈。平台不是把工具交给用户就结束，而是要让可靠能力被发现、复用和持续维护。

## 二、典型问答

### 1. 平台型 AI 产品和普通 AI 应用有什么区别？

**面试官在看：** 你是否知道平台的用户、目标和复杂度不同。

**建议回答：** 普通应用围绕一个终端任务优化输入、输出和体验；平台要让多个团队或多个应用复用能力，因此要处理搭建、测试、发布、权限、版本、成本和治理。平台产品的主指标不只是一次回答是否正确，还包括用户从想法到首次成功的时间、能力复用率、发布回滚效率和下游任务完成。平台不能为了“可配置”牺牲大多数用户的可用性。

**继续追问：** 如果平台用户只有算法和研发，是否还需要重视易用性？

### 2. 设计一个 Agent 搭建平台的核心流程，你会怎么拆？

**面试官在看：** 你是否能把抽象平台落成一条可操作的工作流。

**建议回答：** 我会拆成定义目标和输入、选择模型、配置 Prompt、接入 Skill/Tool、配置知识与 Memory、用真实样例测试、查看失败链路、发布、灰度和线上观察。新用户应能用模板在短时间内完成一个窄任务，专业用户可以继续调参数、看调用轨迹和替换组件。每一步都要有输入输出约束、失败提示和下一步动作，避免平台只提供一堆空白配置表单。

**继续追问：** 你会先做可视化编排，还是先做评估和发布？

### 3. Tool、Skill 和 Agent 为什么要分层？

**面试官在看：** 你是否理解可复用能力和动态执行的边界。

**建议回答：** Tool 解决“能做什么”，Skill 解决“如何完成一类任务”，Agent 解决“面对目标时选择什么并如何继续”。分层后，搜索、数据库查询、发送消息等 Tool 可以被多个 Skill 复用，Skill 也可以被不同 Agent 组合。每层都能独立评估和替换，问题出现时更容易定位。若把所有逻辑放在 Agent Prompt 中，短期灵活，长期会难以复用、测试和治理。

**继续追问：** Skill 和工作流有什么区别？

### 4. 平台配置项越多越好吗？

**面试官在看：** 你是否有抽象和易用性的取舍意识。

**建议回答：** 配置越多不等于能力越强，可能带来学习成本、错误概率、支持成本和版本组合爆炸。我会把配置分成默认安全项、任务常用项和高级项：模板提供合理默认值，用户只在确有需要时进入高级配置；复杂依赖由系统校验并说明影响。平台的价值是让用户更快完成可用任务，而不是把所有内部参数暴露出来。

**继续追问：** 专业用户要求所有参数都可控，如何兼容？

### 5. 如何降低用户从“不会用”到“用起来”的成本？

**面试官在看：** 你是否关注首次成功和学习路径。

**建议回答：** 我会提供按任务组织的模板、示例输入、默认模型和即时预览，让用户先完成一次成功任务，再逐步理解 Prompt、Tool 和知识配置。错误提示要告诉用户为什么失败、应该改什么，而不是返回一串技术日志。新手入口保持简单，专业用户可以打开高级模式、对比版本和查看调用轨迹。首个成功任务比平台宣传的功能数量更能决定留存。

**继续追问：** 用户可以复制模板，但复制后效果很差，你如何排查？

### 6. 如何设计平台的角色和权限？

**面试官在看：** 你是否能同时考虑协作效率和数据安全。

**建议回答：** 我会把创建、编辑、测试、发布、查看线上数据、调用外部 Tool 和管理密钥分开，再按项目、团队和环境隔离。开发者可以在测试环境调试，生产发布需要审批或更高权限，敏感知识和用户数据还要做访问范围控制。所有关键操作保留审计，既方便追责，也方便定位版本变化。权限设计应和业务风险绑定，不能所有人共用一个管理员角色。

**继续追问：** 如果权限过细导致团队不愿意使用，你如何简化？

### 7. 为什么 Prompt、模型、知识库和 Tool 都需要版本管理？

**面试官在看：** 你是否理解 AI 系统行为由多个依赖共同决定。

**建议回答：** 任何一个依赖变化都可能改变结果。只记录 Prompt 版本而不记录模型、知识、Tool 或路由策略，出现 bad case 时无法复现。发布时应保存完整依赖快照，评估通过后再灰度，并允许回滚到上一套组合。版本管理不只是历史记录，还要支持差异比较、负责人、变更原因和影响范围。

**继续追问：** 知识库每天更新，是否要每天生成一个 Agent 版本？

### 8. 如何设计平台的评估能力？

**面试官在看：** 你是否知道平台必须让用户证明效果，而不是凭感觉发布。

**建议回答：** 先让用户导入或创建真实样例，标记正常、边界和风险场景，并定义字段完整、事实正确、引用有效、拒答合理等判断标准。每次改动自动回放固定集，展示版本差异和失败样例；人工评价结果可以补充自动指标。线上反馈经过确认后回流，但不能把模型自己生成的答案直接当成标准答案。评估必须和发布门槛、灰度和回滚连接起来。

**继续追问：** 用户不愿意维护评估集，平台怎样降低成本？

### 9. 如何评价一个 Prompt 或 Agent 平台？

**面试官在看：** 你是否能避免用虚荣指标评价平台。

**建议回答：** 我会看从创建到首次成功的时间、模板使用后的成功率、评估覆盖率、版本迭代周期、发布失败率、回滚时间、组件复用率和下游任务完成率。还要看平台是否降低了重复开发和调试成本。创建了很多 Prompt 但没人发布、没人复用，不能说明平台有价值。对于企业平台，还要加上权限违规、敏感数据暴露和单位有效任务成本等护栏。

**继续追问：** 使用量增长但复用率下降，说明什么？

### 10. 平台如何支持多模型？

**面试官在看：** 你是否能处理效果、成本、延迟和供应商差异。

**建议回答：** 先抽象统一的输入输出 schema、能力标签、错误码和日志，再根据任务要求选择模型。简单任务可用低成本模型，复杂或高风险任务使用更强模型；涉及隐私的任务限定部署范围。平台要用固定评估集展示不同模型的质量、延迟和成本，而不是只提供一个下拉框。模型切换后要重新测试 Prompt、Tool 和上下文长度，不能假设完全兼容。

**继续追问：** 模型 A 质量更好但成本是 B 的三倍，你怎么路由？

### 11. 如何设计 Tool Calling 的产品配置？

**面试官在看：** 你是否关注工具可执行性和失败安全。

**建议回答：** 每个 Tool 要有清晰描述、参数 schema、权限范围、超时、重试、失败返回和示例。产品上要区分只读查询、可修改数据和不可逆动作，并设置确认门槛和调用预算。调用轨迹要能让搭建者看到模型为什么选择该 Tool、参数是什么、结果是否有效。Tool 失败时要给 Agent 可理解的错误，也要给用户可处理的降级路径。

**继续追问：** 模型反复调用同一个失败 Tool，你会在哪里解决？

### 12. Memory 在平台型产品中如何设计？

**面试官在看：** 你是否理解记忆的价值、风险和治理成本。

**建议回答：** 先区分当前任务上下文、会话级摘要和跨任务长期记忆，再定义来源、有效期、可见范围、写入条件、删除和纠错。对用户偏好、稳定事实和训练进度可以考虑长期保留，但敏感信息、未经确认的模型推断和一次性内容不能自动写入。平台需要让用户或管理员知道记忆从哪里来、影响了什么，并支持禁用和删除。

**继续追问：** 如果用户删除记忆，已经生成的历史结果要不要一起删除？

### 13. 如何设计企业知识库和 RAG 的管理能力？

**面试官在看：** 你是否知道 RAG 产品的核心不只是上传文件。

**建议回答：** 知识库要有来源、负责人、权限、版本、生效和失效时间，并支持解析质量检查、分段、检索测试和引用展示。回答前要判断知识是否足够，回答中要保留来源，知识缺失时要拒答或请求补充。更新后必须回放关键评估集，避免新文档覆盖旧规则却无人察觉。不同 Agent 只能检索有权限且与任务相关的知识。

**继续追问：** 检索召回了很多内容，但答案质量仍然低，你怎么定位？

### 14. 如何设计开发、测试和生产环境？

**面试官在看：** 你是否能把试验和线上稳定性隔离。

**建议回答：** 开发环境允许快速改 Prompt 和 Tool，测试环境使用固定评估集、脱敏数据和可重复配置，生产环境强调审批、权限、监控和回滚。环境之间要通过版本发布迁移，而不是直接复制一份“当前配置”。发布包包含完整依赖快照，测试通过后再进行小流量灰度，出现问题可以快速恢复。

**继续追问：** 研发希望直接在生产调 Prompt 验证，你如何处理？

### 15. 平台如何做可观测性？

**面试官在看：** 你是否能从结果追溯到整条 AI 链路。

**建议回答：** 每次任务要关联用户、Agent 版本、模型、Prompt、检索、Tool 调用、延迟、Token、错误、人工修改和最终结果。产品上提供从业务 bad case 下钻到具体链路的方式，并支持按版本、场景、团队和风险筛选。这样才能判断问题出在输入、路由、检索、模型、工具还是展示，而不是只看到一个总体成功率。

**继续追问：** 日志记录越完整越好么？如何兼顾隐私和成本？

### 16. 如何管理平台成本？

**面试官在看：** 你是否会把成本治理产品化。

**建议回答：** 提供按团队、应用、任务、模型和时间的调用量、Token、延迟和费用视图，设置预算、限额和异常告警。优化手段包括缓存重复结果、裁剪无关上下文、模型分级、批处理和失败重试上限。成本优化不能只看单次调用，而要看每个有效完成任务的成本，并用质量和风险护栏防止“便宜但不可用”。

**继续追问：** 某团队成本最高，但它带来的业务价值也最高，如何评价？

### 17. 什么时候平台应该提供模板，什么时候开放自由配置？

**面试官在看：** 你是否能平衡标准化和灵活性。

**建议回答：** 高频、共性、风险清楚的任务优先模板化，给出默认 Prompt、Tool、知识和评估集；差异化或探索性任务可以开放更多配置，但仍保留 schema、权限和发布门槛。模板不是限制，而是把可靠经验前置。平台也要允许从模板复制后形成独立版本，避免一个模板更新影响所有生产 Agent。

**继续追问：** 模板很多导致用户不会选，你会怎么收敛？

### 18. 如何判断一个能力应该平台化，还是只做成单点应用？

**面试官在看：** 你是否会控制平台化冲动。

**建议回答：** 先看需求是否在多个场景重复，输入输出是否可以抽象，是否有稳定的权限和评估方式，以及复用收益能否覆盖平台建设成本。如果只有一个场景、规则仍在快速变化，先做单点应用可能更快验证；等相同能力出现多个稳定需求，再沉淀 Tool、Skill 或共享服务。平台化不是技术荣誉，而是复用和治理的商业决策。

**继续追问：** 业务要求立刻统一所有 Agent，你会如何降低迁移风险？

### 19. 多 Agent 编排什么时候有必要？

**面试官在看：** 你是否理解多 Agent 的收益和复杂度。

**建议回答：** 只有当任务确实需要不同角色、工具或上下文隔离，并且单 Agent 难以稳定完成时，才考虑多 Agent。先定义每个 Agent 的职责、输入输出、交接条件、失败和预算，再用固定样例比较单 Agent 与多 Agent 的任务完成、延迟、成本和可解释性。若只是把一个 Prompt 拆成多个节点，却没有质量收益，多 Agent 只会增加调试和故障面。

**继续追问：** 多 Agent 的某个子节点结果不稳定，平台如何定位？

### 20. 如何用 Autumn PM Coach 证明你理解平台型 AI 产品？

**面试官在看：** 你是否能从自己的项目抽象出平台能力，而不是硬贴概念。

**建议回答：** Autumn PM Coach 的核心不是一个聊天窗口，而是一条可配置的面试成长流程：JD 画像、问题生成、记录整理、证据诊断、训练动作和长期验证。可以把 JD 解析、研究、诊断和成长报告看成不同 Skill，把确定性评分、证据校验、验证场匹配看成受约束的工具或系统能力，把本场输入和历史记忆区分成不同上下文。产品上没有让模型直接写长期记忆或招聘结论，而是让 Agent 负责理解，系统负责校验、持久化和可追溯。这体现了平台产品中“灵活性”和“可信边界”之间的取舍。

**继续追问：** 如果要把这个项目做成平台，你会先抽象哪一层，为什么？

### 21. 平台的抽象边界应该如何确定？

**面试官在看：** 你是否会避免为了“平台化”而过早抽象。

**建议回答：** 先观察多个真实场景中哪些输入输出、权限和评估方式稳定复用，再抽象成 Tool、Skill 或共享服务；差异很大的部分先保留在应用层。抽象要带来复用、替换或治理收益，否则只是增加配置。对 Autumn PM Coach，可以先抽象证据校验、结构化诊断和验证状态，而不是一开始把所有面试流程都做成通用编排器。

**继续追问：** 只有两个业务场景重复，是否值得平台化？

### 22. 如何避免底层模型变化影响上层 Agent？

**面试官在看：** 你是否理解能力契约和兼容性。

**建议回答：** 用统一输入输出 schema、错误码、能力标签和版本快照隔离底层差异；对模型切换重新跑固定评估集和关键场景，不假设 Prompt 完全兼容。平台还要记录模型变化对延迟、成本和行为的影响，必要时支持版本锁定和按任务灰度。上层只依赖稳定契约，不直接依赖供应商的非标准细节。

**继续追问：** 新模型不支持旧模型的某个能力，平台如何处理？

### 23. 如何设计平台的依赖影响分析？

**面试官在看：** 你是否能让用户知道一次改动会影响什么。

**建议回答：** 为 Agent 保存模型、Prompt、Skill、Tool、知识库、Memory 和路由的依赖图。用户修改组件时显示被影响的 Agent、评估集和生产版本，发布前检查权限、schema 和知识有效期。对高风险变更要求重新评估或审批，低风险变更可以自动灰度。影响分析让平台从配置中心变成可治理的产品。

**继续追问：** 依赖图很复杂，用户不愿意看，如何呈现？

### 24. 多租户平台如何做数据隔离？

**面试官在看：** 你是否把权限落实到检索、记忆和日志。

**建议回答：** 按组织、项目、环境和角色隔离知识、Memory、Tool、密钥、日志和费用；模板可以共享，但实例数据默认不共享。检索时做权限过滤，Trace 中对原文和敏感字段做分级展示，导出和复制也要经过权限检查。隔离要用跨租户访问测试验证，不能只依赖页面上看不到入口。

**继续追问：** 一个公共知识库被多个团队使用，怎样避免误共享？

### 25. 平台如何管理 Tool 的生命周期？

**面试官在看：** 你是否考虑 Tool 从接入到废弃的全周期。

**建议回答：** 每个 Tool 需要 owner、用途、输入输出 schema、权限范围、版本、调用限制、错误码、监控和弃用时间。读操作和写操作分级，高风险动作默认需要确认或审批。版本升级要有兼容说明和迁移提示，废弃前查看影响的 Agent 并提供替代方案，避免一个 Tool 下线让大量流程同时失效。

**继续追问：** Tool owner 离职或团队解散，平台怎么办？

### 26. 如何把评估能力做成用户愿意使用的产品？

**面试官在看：** 你是否能降低评估成本，而不是要求用户写复杂测试报告。

**建议回答：** 让用户从真实任务一键保存样例，提供常用评分模板、标注协作、版本对比和失败原因归类。评估结果要能直接关联到下一步动作，例如补 Prompt 约束、增加知识、修复 Tool 或调整阈值。平台还可以从线上 bad case 推荐加入回归集，但必须让用户确认标准答案和敏感信息范围。

**继续追问：** 用户只想看一个总分，不想看失败样例，你如何引导？

### 27. 发布一个 Agent 前至少需要哪些门槛？

**面试官在看：** 你是否能定义安全的发布流程。

**建议回答：** 至少确认目标任务、输入输出 schema、权限、Tool 失败策略、关键评估集、成本和延迟预算、人工接管、日志和回滚点。高风险 Agent 还要经过审批和灰度。发布门槛不必对所有场景相同，但必须与风险等级绑定，不能只根据“演示效果不错”上线。

**继续追问：** 探索性 Agent 没有稳定评估集，能否发布给内部用户？

### 28. 如何设计灰度和回滚？

**面试官在看：** 你是否理解回滚需要恢复完整状态。

**建议回答：** 以 Agent 依赖快照为单位创建发布版本，先在内部或小流量用户中观察任务成功、错误、延迟、成本和风险。灰度期间保留旧版本并设置自动告警，超过阈值时切回整套旧快照。回滚后保留失败样例和 Trace，修复后重新评估，不能直接在生产继续试错。

**继续追问：** 只有模型供应商变了，为什么不能只回滚 Prompt？

### 29. 如何处理 Trace 中的敏感数据？

**面试官在看：** 你是否会平衡调试和隐私。

**建议回答：** 按字段和角色做脱敏、掩码、采样和保留期限控制，调试者默认看到结构化状态和错误类别，而不是所有原文。高权限人员在必要时临时查看，并保留审计。产品还要区分训练、问题排查和合规留存三种目的，不能因为“以后可能有用”就永久保存全部数据。

**继续追问：** 为了定位问题必须查看原文时如何授权？

### 30. Memory 如何支持查看、纠正和删除？

**面试官在看：** 你是否把 Memory 当成可治理数据，而不是神秘能力。

**建议回答：** 每条记忆显示内容、来源、时间、适用 Agent、影响范围和状态，支持用户或管理员确认、修改、过期和删除。删除时要明确是否从后续上下文、索引、摘要和缓存中移除。未经确认的推断只能作为临时上下文，不能默认升级为长期事实。

**继续追问：** 记忆来源是一次聊天，用户又没有主动确认，是否可以保留？

### 31. 平台如何判断一个模板质量好不好？

**面试官在看：** 你是否会用真实复用和任务结果评价资产。

**建议回答：** 看模板被发现、复制、首次成功、重复使用、修改、失败和下线的情况，并按任务和用户群分层。模板还要有版本、owner、适用范围、评估集和风险标签。使用量高但复制后经常失败，说明模板宣传强但质量弱，需要补充示例、限制场景或下架。

**继续追问：** 低使用量但高成功率的模板要不要下线？

### 32. 平台如何支持多 Agent 调试？

**面试官在看：** 你是否能定位跨节点错误和成本。

**建议回答：** Trace 要展示每个 Agent 的输入输出、交接字段、工具调用、耗时、Token、失败和最终结果，并支持回放某个节点。平台还要给每个节点设置预算、最大步数和权限，避免错误在多个 Agent 间扩散。先能解释一条链路，再谈自动优化编排。

**继续追问：** 多 Agent 最终结果错了，但每个节点单看都像正确，怎么办？

### 33. 平台的核心指标应该如何分层？

**面试官在看：** 你是否不会用调用量包装平台价值。

**建议回答：** 采用层看首次成功时间、激活率、复用和留存；能力层看评估通过率、任务完成和错误回归；工程层看延迟、可用性、成本和回滚；治理层看权限违规、敏感数据、未维护组件和高风险发布。最终还要抽样连接下游业务结果，判断平台是否让业务更快、更稳地完成任务。

**继续追问：** 平台调用量下降，但有效任务率上升，是好事还是坏事？

### 34. 什么时候平台能力应该下沉到单点应用？

**面试官在看：** 你是否知道平台不是所有问题的终点。

**建议回答：** 如果用户任务明确、流程固定、配置差异少，直接做应用能提供更好的体验和更少的错误。平台保留通用能力和治理，应用负责场景交互和业务流程。平台产品经理要允许能力以 SDK、组件或受控 API 下沉，而不是为了增加平台功能让终端用户承担复杂度。

**继续追问：** 业务团队希望绕过平台自己接模型，你如何判断是平台问题还是合理自治？

### 35. 如何用 Autumn PM Coach 证明你理解平台的“可信边界”？

**面试官在看：** 你是否能把灵活的 Agent 和确定性系统结合起来。

**建议回答：** 项目中模型可以解析 JD、生成问题、归纳本场文本和提出训练建议，但不能直接写入验证结果、伪造面试证据或把搜索资料混入评分。系统通过 schema、证据校验、固定评分、验证场匹配和 Memory 投影把关键事实收回来。这个设计体现了平台产品的边界：让模型提供灵活性，让系统掌握可信度、权限、持久化和可追溯性。

**继续追问：** 如果用户要求平台允许模型直接改写所有结构化字段，你会开放哪些、限制哪些？

## 三、准备清单

- 能分别从搭建者、使用者、管理员和研发解释平台需求；
- 能讲清 Tool、Skill、Agent、Prompt、Memory、RAG 的边界和依赖；
- 准备一套评估集、版本、发布、灰度、监控和回滚方案；
- 能解释为什么不是所有需求都值得平台化；
- 熟悉权限、审计、数据隔离、成本和供应商切换的基本决策；
- 练习用“首次成功时间、复用率、有效任务成本、质量和风险”替代虚荣指标；
- 能把 Autumn PM Coach 映射成一套有约束的 AI 工作流，而不是泛泛地说“用了 Agent”。
