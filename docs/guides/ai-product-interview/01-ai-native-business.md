# AI 产品经理面试准备一：大厂 AI Native 业务

## 这类岗位到底在考什么

AI Native 岗位不是把原来的搜索框、输入框或推荐位接上一个模型。它要求产品经理从用户目标重新设计任务：哪些步骤由模型理解，哪些步骤由系统执行，哪些结果必须让用户确认，失败时如何回退。

面试官通常不会因为你能背出 RAG、Agent、Function Calling 就给高评价。他们更关心五件事：

1. 你能不能判断一个真实任务为什么需要 AI，而不是为了 AI 找场景。
2. 你能不能把“智能”拆成可执行的输入、判断、工具、结果和兜底。
3. 你能不能接受模型不稳定，并把不确定性设计进产品。
4. 你能不能用任务完成、用户行为和业务结果评价产品，而不是只报模型准确率。
5. 你能不能从 bad case 反推产品、数据、Prompt 和模型应该怎么改。

## 一、基础知识：面试中要讲到什么程度

### 1. AI Native 产品

AI Native 产品把模型的理解、生成、规划或执行作为任务闭环的核心能力。判断标准不是页面上有没有 AI 标志，而是：如果移除模型，用户任务是否会明显退化，产品流程是否需要重新设计。

回答时不要只说“模型是核心”。要补充三层：模型负责什么、系统负责什么、用户在哪些节点拥有控制权。例如 AI 搜索不是把十条搜索结果改成一段话，而是要解决意图理解、检索、证据组织、追问和结果核验。

### 2. Agent、Workflow 和 Chatbot

- **Chatbot**：主要围绕当前输入生成回复，重点是对话理解和回答质量。
- **Workflow**：步骤和分支大部分由系统预先规定，稳定、可测试、容易追责。
- **Agent**：围绕目标动态选择步骤、上下文和工具，并根据执行结果继续判断。

Agent 不是更长的 Prompt。一个场景是否需要 Agent，要看任务是否有动态路径、工具协同和中间结果验证，同时看错误成本是否可接受。规则清楚、路径固定的任务，Workflow 往往比 Agent 更可靠。

### 3. Tool、Skill、Memory 和 Context

- **Tool** 是可执行的原子能力，如搜索、查询订单、发消息或调用业务接口。
- **Skill** 是围绕一类任务沉淀的做法、约束和验收标准。
- **Context** 是当前任务需要的输入、历史对话、检索结果和工具结果。
- **Memory** 是被系统保留下来、预计会对未来任务持续有用的信息，应该有来源、范围、有效期和删除方式。

产品经理要能回答“为什么保留这段信息、谁能看、多久有效、错了怎么改”，而不是把所有聊天记录都称为记忆。

### 4. RAG、Fine-tuning 和 Prompt

- **Prompt** 解决任务说明、角色约束、输出格式和当前上下文问题。
- **RAG** 解决模型需要引用外部、更新频繁或业务私有知识的问题。
- **Fine-tuning** 更适合稳定的格式、风格、分类边界或行为模式，不是更新知识的默认方案。

判断顺序是：模型是不知道信息，还是不会按要求完成任务？需要最新事实和来源时优先考虑 RAG；需要稳定行为时再看 Prompt、工作流、训练或微调的组合。

### 5. Grounding 与幻觉

Grounding 指输出能够落回可核对的输入、知识来源或工具结果。幻觉不只是“说错一个事实”，还包括把没有发生的动作说成已完成、把相关性说成因果、把不确定结论说成确定结论。

产品手段包括引用来源、展示执行状态、结构化输出、事实校验、置信度分层、人工确认和明确的“不知道”。高风险场景不能靠一句“请模型不要编造”解决。

### 6. AI 评估的层级

建议按四层回答：

1. **能力层**：分类、抽取、召回、生成质量是否达到基线。
2. **任务层**：用户是否完成了目标，关键步骤是否正确。
3. **体验层**：采纳、修改、放弃、响应时间和人工接管如何。
4. **业务层**：转化、留存、效率、成本或风险是否改善。

模型离线分数高但任务完成率低，通常说明评估集、流程或交互没有覆盖真实问题。

### 7. 任务成功和回答质量不是一回事

回答质量是局部判断，任务成功是用户是否完成了目标。例如回答很完整，但没有帮用户做出选择、提交申请或解决问题，仍然不能算任务成功。AI 产品要先定义任务终点，再把回答正确性、可执行性和用户后续动作连接起来。

### 8. 不确定性与拒答

模型不确定时，产品可以要求补充信息、展示候选答案、引用依据、转人工或明确拒答。拒答不是能力不足，而是风险控制的一部分。好的拒答应该说明缺少什么、为什么不能继续、用户下一步可以做什么。

### 9. Tool 的副作用安全

查询类 Tool 通常是可逆和低风险的，写入、发送、购买、删除和发布类 Tool 会产生外部副作用。产品要区分预览、确认、执行和回执状态，配合权限、参数校验、幂等、审计和回滚，不能只依赖模型“自觉谨慎”。

### 10. Agent 状态和检查点

Agent 执行任务时需要记录当前目标、已完成步骤、工具结果、待确认事项和失败原因。检查点可以支持继续执行、人工接管、重试和回滚，也便于定位“模型说完成了但系统其实没完成”的状态错误。

### 11. 人工评估的一致性

生成式任务经常没有唯一答案，因此需要评分指南、正反例和多人标注一致性。评估标准不能只写“回答好不好”，而要拆成事实、完整性、相关性、可执行性、引用和风险等维度，并说明哪些错误是致命错误。

### 12. 模型路由的产品决策

模型路由可以按任务难度、风险、隐私、时延和成本选择不同模型。路由不是简单的“复杂问题用大模型”，还要有可回放的规则、评估门槛和失败降级，避免用户得到不可解释的质量波动。

### 13. 数据飞轮不是自动发生的

用户反馈只有经过采集、清洗、分类、人工确认和版本化，才可能成为数据资产。点击、点赞和修改不一定代表正确答案，必须结合最终任务结果和业务判断，避免把噪声反馈直接用于优化模型或记忆。

### 14. 个性化与用户控制

个性化可以提高相关性，但也可能造成过度推断、旧偏好污染和信息茧房。产品应区分用户明确提供的偏好、系统从行为推断的倾向和一次性上下文，并提供查看、修改、删除和关闭个性化的入口。

### 15. AI 交互的状态表达

Agent 产品需要让用户理解当前处于输入、规划、检索、调用工具、等待确认、完成或失败哪一种状态。状态表达不能把“正在生成”伪装成“已经完成”，也不能只给一个转圈。进度、证据、可取消和可恢复会直接影响用户信任。

### 16. AI 产品的长期壁垒

模型本身容易被替换，长期壁垒更可能来自真实任务数据、评估集、工作流、领域知识、用户反馈和组织协作方式。产品经理需要证明这些资产如何沉淀并改善任务结果，而不是把“接入某个模型”当成壁垒。

## 二、典型问答

### 1. 你如何判断一个需求适合做 AI Native 产品？

**面试官在看：** 你是否从用户任务和价值出发，而不是从模型能力出发。

**建议回答：** 我会先描述用户要完成的结果、当前流程和主要成本，再判断其中是否存在理解、生成、搜索、规划或执行的高成本环节。适合 AI 的任务通常有较多非结构化输入、需要结合上下文或存在个性化判断，但还必须满足两个条件：结果可以被验证，错误可以被发现或被人工兜底。最后用一个小场景验证任务完成时间、结果质量和用户采纳，而不是先做万能助手。

**继续追问：** 如果用户只是想查一个确定答案，为什么不用普通搜索或规则？

### 2. 什么时候应该用 Workflow，什么时候才用 Agent？

**面试官在看：** 你是否理解可控性和灵活性的取舍。

**建议回答：** 如果步骤稳定、分支有限、验收标准明确，我会优先用 Workflow，因为更容易测试、监控和追责。如果用户目标相同但路径会因输入、工具结果和环境变化而变化，才考虑 Agent。即使使用 Agent，我也会把高风险步骤固定化，把工具权限、最大步数、超时和人工确认设为系统约束。Agent 解决的是动态决策，不是替产品经理逃避流程设计。

**继续追问：** 一个 Agent 连续调用工具失败时怎么办？

### 3. 如何从用户目标拆一个 Agent？

**面试官在看：** 你能否把抽象概念落到产品架构。

**建议回答：** 我会按“目标、输入、计划、工具、状态、验证、输出、接管”拆解。先定义完成标准，例如“找到满足约束的三件商品并说明差异”，再决定需要哪些信息和工具。每个工具写清参数、权限和失败返回；每一步产生结构化状态；最终输出前检查是否满足目标。用户需要知道 Agent 做了什么、依据是什么、下一步是否需要确认。

**继续追问：** 为什么不能直接让大模型自己决定所有步骤？

### 4. 大厂 AI Native 产品的 MVP 应该怎么定？

**面试官在看：** 你能否控制范围并建立可验证闭环。

**建议回答：** 我会选择一个高频、边界清楚、错误成本可控、结果可度量的核心任务。例如先验证“根据用户约束完成一次可解释的对比”，而不是同时覆盖搜索、推荐、售后和复购。MVP 需要明确不做什么、人工如何兜底、主指标是什么、达到什么结果才扩展。先证明用户愿意把任务交给系统，再增加 Agent 自主性。

**继续追问：** 如果业务坚持第一版就做全链路怎么办？

### 5. 如何定义 AI 产品的主指标？

**面试官在看：** 你是否把模型指标连接到用户任务。

**建议回答：** 先确定用户是否完成目标，再选任务完成率、有效采纳率或问题解决率作为主指标。然后配质量、延迟、成本、人工接管和风险指标作为护栏。例如购物助手不能只看回答准确率，还要看用户是否找到合适商品、是否减少重复咨询，以及推荐错误是否带来投诉。指标必须写清分母、时间窗口和成功条件。

**继续追问：** 用户点了“采纳”但最后没有完成业务动作，算成功吗？

### 6. 模型回答很流畅，但用户不采纳，你怎么判断问题？

**面试官在看：** 你是否能拆解“质量差”的不同原因。

**建议回答：** 我会把漏斗拆成理解、生成、查看、采纳、修改和最终任务结果。用户不采纳可能是事实不可信、缺少来源、答案不可执行、入口不合适、响应太慢，也可能是用户根本不需要这类帮助。先用日志、用户访谈和 bad case 分层，确认是模型问题、产品问题还是场景价值问题，再决定改 Prompt、知识、交互还是直接下线。

**继续追问：** 如果用户不采纳但停留时间变长，如何解释？

### 7. 如何处理 Agent 的幻觉和错误动作？

**面试官在看：** 你是否能把风险产品化。

**建议回答：** 先按错误成本分层。低风险的内容可以给来源和纠错入口；涉及下单、支付、发帖、删除和用户权益的动作必须预览并确认。工具调用要有权限、参数校验、幂等、超时和回滚；输出要区分“建议”“已查询”和“已执行”。无法验证时必须降级为不确定或转人工，不能用更强语气掩盖不确定性。

**继续追问：** 置信度低一定要转人工吗？

### 8. RAG 和 Fine-tuning 如何选择？

**面试官在看：** 你是否能从问题性质做技术产品判断。

**建议回答：** 我会先判断问题是知识缺失、知识更新、检索不准，还是行为和格式不稳定。需要引用最新知识时优先 RAG，并评估切分、召回、重排和引用质量；需要稳定输出格式或特定分类行为时考虑 Prompt、结构化约束或微调。二者也可以组合，但要先用最轻量方案验证瓶颈，避免把检索问题误诊为模型能力问题。

**继续追问：** RAG 加入后答案仍然错，先查哪里？

### 9. Memory 应该保存什么，不应该保存什么？

**面试官在看：** 你是否理解长期记忆的价值和治理成本。

**建议回答：** 只保存未来任务有稳定价值、来源明确、用户可接受的信息，例如长期偏好、已确认的工作规则或持续目标。当前任务的临时上下文应放在短期上下文，不应自动升级为长期记忆。每条记忆要有来源、时间、适用范围、置信度、修改和删除入口；冲突时优先新确认的信息，不能让旧记忆静默覆盖用户当前意图。

**继续追问：** 用户说“这次先这样”，为什么不能记住？

### 10. 上下文越来越长，如何控制效果和成本？

**面试官在看：** 你是否会做上下文工程，而不是只加 Token。

**建议回答：** 先按任务价值筛选上下文，区分必须事实、可选参考和历史噪音。对长记录做结构化摘要，但保留来源和时间；对检索结果限制数量并做去重；对重复工具结果做缓存。压缩后仍要用代表性样例检查关键事实是否丢失，不能为了省成本把决定性证据摘要掉。

**继续追问：** 摘要把关键否定条件丢了怎么办？

### 11. 多 Agent 什么时候有必要？

**面试官在看：** 你是否会抵抗过度架构。

**建议回答：** 只有当任务角色、工具权限、评估标准或上下文边界确实不同，拆分才有价值。例如规划 Agent 负责拆任务，执行 Agent 负责调用业务工具，审核 Agent 负责风险检查。否则多个 Agent 会增加通信、成本和错误传播。第一版通常先用单 Agent 加固定 Workflow 验证，只有单体结构已经难以维护或评估时再拆。

**继续追问：** 多 Agent 之间如何避免互相传错信息？

### 12. 如何设计人工接管？

**面试官在看：** 你是否把人工当作流程角色，而不是错误垃圾桶。

**建议回答：** 设定触发条件：高风险动作、低置信度、关键工具失败、用户明确不满意或连续重试。接管界面需要展示用户目标、已完成步骤、依据、失败原因和建议处理方式，并让人工能够修改后继续执行。还要记录人工修改和最终结果，区分是模型错、规则错还是流程配置错。

**继续追问：** 人工接管率越低越好吗？

### 13. 如何搭建 AI 产品的评估集？

**面试官在看：** 你是否理解评估不是随便打分。

**建议回答：** 从真实任务抽取正常、边界、长尾和高风险样例，并按场景、用户类型和错误成本分层。每条样例定义可判断的成功标准，必要时允许多个合理答案。评估集版本化，线上 bad case 经人工确认后再回流；离线评估用于比较版本，线上任务指标用于验证真实价值，二者不能互相替代。

**继续追问：** 生成式答案没有唯一标准，怎么评估？

### 14. 任务完成率突然下降，你如何排查？

**面试官在看：** 你是否有系统化故障定位能力。

**建议回答：** 先确认指标口径、埋点和样本量，排除统计错误。再按模型版本、Prompt、输入分布、检索质量、工具成功率、延迟和外部业务变化分层对比。短期先回滚、降低自动化范围或增加人工兜底；确认根因后再修复，最后补一个能提前发现同类问题的评估样例或监控。

**继续追问：** 只在某个用户群下降，如何定位？

### 15. 如何建立 bad case 到迭代的闭环？

**面试官在看：** 你是否能让模型产品持续变好。

**建议回答：** 先记录完整链路：输入、上下文、模型版本、工具调用、输出、用户动作和最终结果。再给错误分类，如意图理解、知识缺失、检索、规划、工具参数、表达或产品流程。按频率、影响和修复成本排序，形成评估集和回归测试；每次改动都要确认旧问题改善且没有明显回归。

**继续追问：** 用户修改答案能直接作为正确答案吗？

### 16. 多模态能力什么时候值得做？

**面试官在看：** 你是否能判断真实增量，而不是追热点。

**建议回答：** 当任务信息天然分布在图片、视频、语音和文本中，或单纯文本会损失关键上下文时，多模态有价值。例如识别商品图片、理解直播画面或分析面试语音。要比较加入多模态后任务完成率、错误率、延迟和成本的变化；如果只是把图片转成文字再走原流程，可能不值得承担额外复杂度。

**继续追问：** 多模态结果错误时如何让用户纠正？

### 17. AI 产品如何建立用户信任？

**面试官在看：** 你是否理解信任来自控制和可解释，而不是宣传。

**建议回答：** 对事实给来源，对动作给状态和预览，对不确定结论给置信度或备选方案。用户应该能修改、拒绝、撤销和转人工。还要保持结果稳定，避免同一输入反复变化到无法理解。信任不是让用户相信模型永远正确，而是让用户知道模型哪里可靠、哪里需要自己判断。

**继续追问：** 展示更多解释会不会反而增加用户负担？

### 18. AI 产品如何处理隐私与安全？

**面试官在看：** 你是否知道 AI 产品的风险不只来自模型。

**建议回答：** 在输入侧做敏感信息识别和最小化收集，在上下文和 Memory 侧做权限、范围和生命周期控制，在 Tool 侧做最小权限和用户确认，在输出侧做敏感内容和越权检查。日志需要能定位问题，但不能无边界保存原文。隐私要求应该进入产品流程和验收标准，而不是上线前补一段免责声明。

**继续追问：** 为了提升效果，是否可以默认保存所有对话？

### 19. 如何把模型能力连接到业务价值？

**面试官在看：** 你是否能跨过“模型效果不错”这一层。

**建议回答：** 先写清任务链：模型输出改变了哪个用户动作，用户动作又影响哪个业务指标。例如更准确的意图识别减少了重复咨询，进而提升问题解决率和留存。每一层都要有可观测指标，并注意外部因素和归因限制。不能把调用次数、停留时长或点赞直接等同于价值。

**继续追问：** 业务指标变好但模型指标变差，如何判断？

### 20. 如何介绍自己的 AI Native 项目？

**面试官在看：** 你是否真正理解自己的项目，而不是罗列技术名词。

**建议回答：** 可以用“用户任务 - AI 判断 - 系统约束 - 证据 - 结果”讲秋招助手：它不是实时面试机器人，而是把 JD 转成岗位画像和问题，再把面试记录转成证据型诊断和训练动作。模型负责理解 JD、生成问题和归纳诊断；系统负责字段校验、证据约束、评分归一化和训练验证；长期记忆只由确定性代码聚合。这样能说明你理解 Agent 的边界，也能解释为什么不把搜索资料混成本场表现证据。

**继续追问：** 如果把你的项目变成一个 Agent，最先增加什么能力，为什么？

### 21. 用户目标说得很宽，你如何判断真正的意图？

**面试官在看：** 你是否知道“理解用户”不是把一句话改写得更长。

**建议回答：** 我会把目标拆成期望结果、约束条件、当前状态和下一步动作，并在信息不足时先澄清最影响决策的条件。例如“帮我准备面试”可能需要知道岗位、轮次、JD、经历和时间限制。产品上可以用结构化追问和默认选项减少用户负担，同时记录用户确认过的目标。只有目标明确，后续检索、规划和评价才不会漂移。

**继续追问：** 追问太多会让用户流失，你会优先问什么？

### 22. 如果生成答案没有唯一标准，如何判断版本变好了？

**面试官在看：** 你是否能建立可重复的生成式评估。

**建议回答：** 先把“好”拆成任务相关维度，例如事实准确、关键点完整、引用有效、表达可执行和风险合规，再为每个维度写评分锚点和反例。可以结合规则校验、人工抽评和模型辅助评估，但要定期检查评估器与人工判断的一致性。版本比较时不仅看平均分，还要单独看高风险样例和明显回归。

**继续追问：** 两个评估者意见不一致，你会取平均分吗？

### 23. 模型不知道答案时，产品应该怎样设计？

**面试官在看：** 你是否把“不知道”当成正常状态。

**建议回答：** 先判断能否通过补充信息、检索或工具调用降低不确定性；如果仍不能验证，就明确说明缺少依据，并给出可行的下一步。高风险任务宁愿拒答或转人工，也不应生成一个看似完整的答案。界面上要区分“没有找到资料”“资料之间冲突”“系统暂时失败”和“超出能力范围”，帮助用户做正确决策。

**继续追问：** 用户强行要求给一个确定答案，你会如何回应？

### 24. Tool 被恶意输入诱导时如何防护？

**面试官在看：** 你是否理解 Prompt Injection 和工具越权是产品问题。

**建议回答：** 不能让检索到的文本直接改变系统权限或工具策略。工具需要显式权限、参数 schema、调用范围和敏感操作确认；外部内容作为数据而不是指令处理，输出再经过策略校验。对发送、支付、删除等动作做最小权限和可撤销设计，并记录调用链路。安全验证要纳入固定攻击样例和回归测试。

**继续追问：** 如果攻击文本藏在用户上传的文档里，在哪里拦截？

### 25. 模型升级后整体分数变高，但线上投诉增加，怎么处理？

**面试官在看：** 你是否会识别平均分掩盖的分布变化。

**建议回答：** 先暂停全量发布，按用户群、任务类型、输入长度、语言风格和风险等级比较新旧版本。重点检查长尾错误、拒答行为、工具调用和用户体验，而不是只看平均分。必要时回滚，补充投诉样例和分层指标，再决定是否调整模型、Prompt、路由或发布范围。模型升级必须是可灰度和可回退的产品流程。

**继续追问：** 什么情况下你会接受线上投诉短期上升？

### 26. 质量、延迟和成本互相冲突时怎么取舍？

**面试官在看：** 你是否会按任务价值分层，而不是全局追一个数字。

**建议回答：** 先按任务风险和用户耐心设定最低质量门槛，再在门槛之上比较延迟和成本。高价值或高风险任务可以使用更强模型和更长等待；低风险任务可以使用缓存、轻量模型或异步处理。决策单位最好是“每完成一个有效任务的成本”，并且设置质量、投诉和失败率护栏。

**继续追问：** 用户明确表示更在意快，而不是更准，你会怎么验证？

### 27. Agent 陷入循环或不断重复调用，如何设计？

**面试官在看：** 你是否把运行时边界作为产品能力。

**建议回答：** 设置最大步数、调用预算、相同参数去重、超时和循环检测，并为每一步记录状态。连续失败时不要继续消耗资源，而是总结已完成内容、解释失败原因并转人工或请求补充信息。平台还要能让产品经理看到循环发生在哪个 Tool、哪个输入和哪个版本，避免只靠日志排查。

**继续追问：** 限制步数会不会损失复杂任务的成功率？

### 28. 用户当前意图和历史记忆冲突时听谁的？

**面试官在看：** 你是否理解记忆不是事实真相。

**建议回答：** 当前明确表达的意图优先于历史推断，历史记忆只能作为参考。系统应把记忆来源、时间和适用范围带入判断，遇到冲突时提示用户确认，而不是静默覆盖。对于关键偏好可以提供“本次使用”与“更新长期偏好”的区分，避免一次临时选择污染后续任务。

**继续追问：** 如果用户没有明确说要更新记忆，行为变化能不能自动更新？

### 29. 如何把用户反馈变成可用训练数据？

**面试官在看：** 你是否知道反馈需要治理。

**建议回答：** 先记录上下文、输出、用户动作和最终任务结果，再按事实错误、遗漏、风格、流程和需求不匹配分类。对高影响样例进行人工确认，补充期望结果和适用范围后才进入评估集。用户的“修改过”只说明原结果不够好，不自动等于修改后的内容是唯一正确答案。

**继续追问：** 用户修改内容和业务规则冲突时如何处理？

### 30. 用户规模突然增长，AI 产品如何保证可用？

**面试官在看：** 你是否会关注容量、成本和降级。

**建议回答：** 先监控请求量、队列、延迟、错误率、Token 和供应商限流，再按任务重要性分级。可以通过缓存、排队、异步结果、轻量模型、限额和人工兜底保护核心任务；明确哪些功能可以暂时关闭，哪些必须保持。扩容不是唯一方案，产品需要为高峰时的用户预期和失败反馈设计清楚。

**继续追问：** 哪些任务可以被降级，哪些绝不能降级？

### 31. 如何把内容安全做成 AI Native 产品的一部分？

**面试官在看：** 你是否能把安全从“审核模块”扩展到全链路。

**建议回答：** 从输入、检索内容、模型输出、工具调用和用户反馈五个位置识别风险。输入侧识别敏感和恶意指令，知识侧控制来源和权限，输出侧做策略检测，工具侧限制副作用，线上持续收集新型 bad case。安全策略需要按风险分级，不是所有内容都用同一套拒答，也要给用户可理解的替代路径。

**继续追问：** 安全策略导致正常请求被拒，如何优化？

### 32. 如何判断一个 AI 功能是在解决问题，还是在制造新问题？

**面试官在看：** 你是否有反事实和全流程思维。

**建议回答：** 比较引入 AI 前后完整任务链的时间、错误、返工、投诉、用户控制和组织成本，而不是只看模型输出。一个功能如果减少了输入，却增加了核验、纠错和解释成本，净收益可能是负的。可以让用户选择原流程和 AI 流程进行试点，观察最终结果和长期复用，而非只看第一次体验。

**继续追问：** 如果用户喜欢 AI 结果，但业务成本变高，如何决策？

### 33. 如何按用户群评估 AI 产品？

**面试官在看：** 你是否意识到平均指标会掩盖差异。

**建议回答：** 按新老用户、任务熟练度、输入复杂度、行业或岗位、风险等级和使用频率分层，分别看成功率、采纳、修改、延迟和投诉。新用户可能需要更多引导，专业用户更关注可控性和批量效率。分层结果能帮助决定是做个性化交互、不同模型路由，还是收窄某类用户范围。

**继续追问：** 只在高价值老用户上有效，是否值得继续做？

### 34. AI 功能是否会伤害原有产品行为？

**面试官在看：** 你是否会考虑替代、依赖和业务结构变化。

**建议回答：** 观察 AI 是否减少了有价值的探索、内容生产或人工判断，是否让用户过度依赖错误答案，是否挤压原有核心业务。可以设置透明提示、人工确认、结果来源和多样性护栏，并比较引入前后长期留存和任务质量。AI 不应只追求短期点击，还要考虑用户能力和业务生态是否健康。

**继续追问：** 如果 AI 让旧功能使用下降，但总体业务结果变好，如何判断是正向替代？

### 35. 面试官要求你立刻增加一个复杂 Agent 能力，你如何判断？

**面试官在看：** 你是否能坚持最小可验证原则。

**建议回答：** 先问清它要解决的用户任务、当前失败、预期指标、数据来源、工具权限和风险，再看现有 Workflow、检索或人工流程能否先验证。只有当动态规划带来的成功率或任务价值明显超过固定流程，并且复杂度可观察、可回滚，才进入 Agent 方案。对 Autumn PM Coach，我会优先完善 JD 画像复用、证据诊断和训练验证，而不是先做实时面试或无限制的自主搜索。

**继续追问：** 如果这个复杂能力是管理层指定的战略方向，你如何在不阻塞主线的情况下推进？

## 三、准备清单

- 准备一个能在三分钟内讲清的 AI Native 项目，包含用户、任务、模型边界、指标和 bad case。
- 至少准备一个“模型输出正确但产品仍失败”的例子。
- 能解释一个场景为什么不用普通规则、Workflow 或搜索。
- 能画出 Agent 的输入、上下文、工具、状态、验证和人工接管。
- 能区分离线能力评估、线上任务指标和最终业务结果。
- 不要把“用了 Agent、RAG、多模态”当成项目价值本身。
