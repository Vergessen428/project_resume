# Autumn PM Coach 产品逻辑与实现说明

## 1. 文档目的

本文用于统一说明 Autumn PM Coach 的产品定位、核心流程、Agent 与 Workflow 的分工，以及外部面经、当前面试证据和长期记忆之间的关系。

它回答四个问题：

1. 这个产品到底解决什么问题？
2. JD、简历、面经和面试记录如何形成一条逻辑链？
3. 哪些判断应该交给模型，哪些判断必须由系统确定性完成？
4. 用户如何从一次复盘得到下一次可以验证的训练动作？

## 2. 产品定位

Autumn PM Coach 是一个**以 JD 为主线的产品经理求职面试助手**，覆盖面试前准备、面后复盘、训练提升和长期成长。

它不是单纯的面经搜索器，也不是只输出分数的评分页面。它要建立的是一条可解释的链路：

```text
JD / 简历 / 公开面经
        ↓
岗位画像与考察重点
        ↓
面前问题准备
        ↓
面试转写 / 手写记录 / 面后补充
        ↓
证据型诊断
        ↓
针对性训练动作
        ↓
下一场自动验证
        ↓
长期能力变化
```

产品的核心价值不是替用户判断“能不能拿到 offer”，而是帮助用户理解：

- 为什么这个岗位可能问这些问题；
- 自己的回答实际覆盖了哪些岗位要求；
- 哪些问题是没有证据、回答不完整或逻辑矛盾；
- 下一次应该具体补哪一段回答；
- 这个缺口在后续面试中是否真的改善。

## 3. 总体判断：流程是否合理

原流程的主线是合理的，尤其是把 JD 放在系统中心，并把一次面试连接到训练和长期记忆。但需要做以下修正，才能成为真实可用的产品：

### 3.1 JD 不是搜索输入，而是岗位目标定义

JD 同时服务于五个阶段：

| 阶段 | JD 的作用 |
| --- | --- |
| 面前准备 | 推断岗位可能考察的能力和问题 |
| 资料搜索 | 生成公司、岗位、轮次和主题查询 |
| 面后诊断 | 判断回答覆盖了哪些岗位要求 |
| 训练提升 | 将能力缺口绑定到岗位要求 |
| 长期成长 | 比较不同面试中相同岗位能力的变化 |

因此系统应尽量生成一份统一的 `jd_profile`，后续模块复用，而不是每个模块重新理解一遍 JD。

```json
{
  "role_family": "ai_product",
  "responsibilities": [],
  "requirements": [],
  "core_capabilities": [
    {
      "id": "metrics_experiment",
      "importance": "must_have",
      "evidence_expected": "指标定义、归因和验证"
    }
  ],
  "question_topics": [],
  "search_topics": []
}
```

### 3.2 面经是准备上下文，不是面试事实

公开面经可以帮助用户提前理解岗位语境、常见追问和可能的考察方向，但不能直接证明用户在本场面试中的表现。

必须分开三种资料：

| 资料类型 | 可以做什么 | 不可以做什么 |
| --- | --- | --- |
| 外部研究资料 | 生成准备题、补充岗位语境 | 作为候选人评分证据 |
| 本场面试证据 | 评分、诊断、生成缺口 | 用历史资料替代原文 |
| 长期记忆 | 判断重复问题和训练进展 | 直接改写本场事实 |

这条边界可以避免系统出现“别人面经中答得好，所以用户这一项得分高”的错误。

### 3.3 “没有再次出现”不能直接等于改善

下一场面试没有出现某个 gap，可能只是面试官没有问到。因此验证必须同时满足：

- 后续面试确实覆盖了对应能力；
- 后续面试存在对应技能或子维度的原文证据；
- gap 没有重复，或分数达到确定性提升阈值。

如果后续场次没有覆盖该能力，结果应为 `pending`，而不是 `supported`。

### 3.4 面试结果只能作为描述性反馈

通过或未通过可以记录，但不能反推“某一项能力导致结果”，也不能修改本场评分。

结果反馈用于回答：

- 当前样本是否足够；
- 某类训练动作之后是否出现了结果变化；
- 是否需要继续收集证据。

它不用于生成录用概率，也不用于替代用户或招聘方的判断。

## 4. 用户完整使用流程

### 4.1 面前：建立岗位画像

用户填写公司、岗位、面试轮次、JD，必要时关联一份简历。系统提取：

- 岗位职责和任职要求；
- 关键词和岗位族；
- 重点 PM 能力；
- 预期证据；
- 面试前准备主题；
- 后续搜索查询。

如果没有简历，系统仍然可以根据 JD 生成岗位问题，但会把“经历证据不足”标记为待补充，而不是假设用户有相关经验。

### 4.2 面前：生成针对性问题

问题不应是固定问卷，而应来自 JD、简历和可选面经的交集。建议分为三类：

- **命中题**：JD 要求和简历已有经历的交集；
- **补刀题**：JD 明确重要，但简历中缺少证据；
- **深挖题**：围绕项目、指标、决策、协作和结果继续追问。

每道题需要携带能力标签和生成理由，例如“JD 强调指标归因，但简历只写了增长结果，因此需要补充验证方法”。

这不是实时模拟面试，而是帮助用户在面试前或面试刚结束时快速重建思考过程。

### 4.3 公开资料研究

研究 Agent 根据以下信息生成有限查询：

```text
公司 + 岗位 + 轮次 + JD 主题 + 平台限定词
```

资料处理状态建议为：

```text
候选链接
→ 公开内容已读取
→ OCR / 图片内容已提取
→ AI 已结构化
→ 用户已确认
→ 可用于面试准备
```

小红书资料不应承诺“稳定抓取”。更准确的产品描述是：

> 系统通过公开搜索发现候选链接，并尝试读取公开页面；如果页面只有动态壳或无法访问，则保留链接和失败原因，允许用户补充截图或文本，再进行结构化提取和确认。

小红书资料的自动化可以降低搜索成本，但不应绕过登录、Cookie、访问控制或平台限制。

### 4.4 面后：收集当前面试证据

用户可以输入：

- 面试官原问题；
- 自己的回答；
- 面试官追问；
- 自己没有接住的部分；
- 项目指标、决策、协作和结果；
- 面后补充的记忆。

这些内容共同形成当前面试的 `transcript` 或手写记录。系统需要提醒用户：没有原文证据时，应标记为 `missing` 或 `unverified`，不能让模型补写事实。

### 4.5 证据型诊断

诊断流程应当是：

```text
本场原文
→ 证据片段
→ PM 能力与子维度
→ 评分锚点
→ 缺口与影响
→ 训练动作
```

六项 PM 能力保持不变：

- 产品判断；
- 项目主导力；
- 指标与实验；
- 推进与协作；
- 结构化表达；
- 业务与岗位理解。

每项能力继续拆成可观察的子维度，并明确：

- `observed`：原文有明确证据；
- `missing`：该维度没有足够证据；
- `contradicted`：原文存在相互矛盾的表现；
- `not_applicable`：本场不适用。

评分必须由固定权重和评分锚点归一化，模型负责提取证据和解释，不直接决定最终分数。

### 4.6 训练动作与下一场验证

训练动作必须绑定四项信息：

```text
当前缺口 + JD 要求 + 本场证据 + 下一场验收方式
```

例如：

```text
训练动作：重写一次项目指标回答

验收标准：
- 说明核心指标
- 说明护栏指标
- 解释指标和业务目标的关系
- 给出归因假设
- 说明验证方法
```

下一场验证由后端自动认领：选择日期更晚、已经完成复盘、岗位族兼容，并且公司相同或岗位名称归一化相同的最近场次。

验证结果只有三档：

| 状态 | 含义 |
| --- | --- |
| `supported` | 后续场次实际覆盖能力，gap 未重复且技能分数达到提升阈值，或证据满足训练标准 |
| `partially` | 后续场次实际覆盖能力，但 gap 仍存在或提升不足 |
| `pending` | 没有兼容的后续场次，或后续场次没有覆盖该能力 |

验证结果由确定性代码计算。模型不能直接写入 `validation`，也不能通过面试结果修改它。

## 5. Agent 与 Workflow 的分工

### 5.1 Workflow 负责确定性事实

Workflow 或普通后端代码负责：

- 解析和保存 JD；
- 生成和保存 `jd_profile`；
- 固定能力 ID、子维度 ID 和评分权重；
- 计算资料相关度；
- 校验本场 evidence 是否来自原文；
- 限制字段长度和枚举值；
- 自动认领验证场；
- 计算 `supported / partially / pending`；
- 聚合长期记忆；
- 持久化 outcome；
- 保护用户修正字段。

这些逻辑要可测试、可复算、可追溯。

### 5.2 Agent 负责需要语义判断的部分

最小代码原则下，保留一个 PM Coach Agent，按模式使用不同 Prompt 即可：

1. **准备模式**：根据 JD、简历和研究线索生成命中题、补刀题和深挖题；
2. **研究模式**：改写有限查询，理解公开资料，抽取问题和考察主题；
3. **复盘模式**：从本场文字中提取证据、映射能力并解释缺口；
4. **训练模式**：把缺口改写成训练动作、追问和验收标准。

不需要拆成多个互相调用的 Agent，也不需要在 V3.1 引入复杂 Tool Calling。历史相关记录可以由后端确定性预取受限片段，再注入复盘上下文。

### 5.3 Agent 的输入和输出边界

Agent 可以读取：

- 当前 JD profile；
- 当前简历上下文；
- 当前面试文本；
- 已归一化的能力规则；
- 受限的历史相关片段；
- 已计算的长期记忆摘要。

Agent 不可以直接决定：

- 本场最终分数；
- 证据是否来自本场；
- 搜索资料是否为真实原帖；
- 验证是否成功；
- 面试结果是否由能力导致；
- 数据库中的长期记忆事实。

## 6. 资料相关度如何判断

平台约束是搜索条件，不混入相关度总分。候选资料可以采用固定权重：

| 维度 | 权重 |
| --- | ---: |
| 公司匹配 | 30% |
| 岗位匹配 | 25% |
| 轮次匹配 | 15% |
| 主题匹配 | 15% |
| 面试具体性 | 10% |
| 时效性 | 5% |

每条结果还需要保存：

- 查询词；
- 来源 URL；
- 平台；
- 相关度分解；
- 计算理由；
- 读取状态；
- 用户确认状态；
- 可用于的场景。

相关度高只代表“值得准备时查看”，不代表内容真实，也不代表一定会在面试中出现。

## 7. 长期记忆如何工作

长期记忆不是完整聊天记录，而是由每场结构化复盘投影得到的摘要：

```text
岗位要求
→ 本场证据
→ 当前缺口
→ 训练动作
→ 后续验证
→ 多场重复或改善
```

建议保留三层：

### 单场记录

保留本场原始输入、诊断结果、训练动作和结果反馈。

### 短期训练记忆

保留最近的开放行动、最近验证状态和需要在下一场关注的能力。

### 长期成长摘要

聚合重复缺口、能力趋势、训练验证和岗位族差异。样本不足或评分版本不一致时，不生成强结论。

## 8. 当前产品的边界与风险

### 8.1 不把模型输出当作事实

所有本场证据必须能回指当前输入文本。外部面经、历史片段和模型推断只能进入解释上下文。

### 8.2 不把评分当作招聘结论

训练分只帮助排序训练优先级，不代表录用概率、候选人价值或面试结果。

### 8.3 不承诺自动读取所有小红书内容

公开页面可能存在动态渲染、访问限制、内容缺失或链接失效。系统必须展示失败原因和来源状态，而不是静默显示“没有资料”。

### 8.4 不因一次面试生成长期结论

单场结果只能说明当前场次的证据情况。只有多场、同一评分版本、相对稳定的岗位上下文，才适合讨论趋势。

### 8.5 不过早引入复杂 Agent 架构

当前单次模型调用加确定性预取已经可以覆盖核心体验。只有当真实使用证明“固定预取无法满足历史追问”时，才考虑工具调用循环。

## 9. 当前版本与后续实现重点

### 当前可展示能力

- JD 岗位画像和针对性问题；
- 六项 PM 能力的证据型诊断；
- 训练动作和验收标准；
- 三档训练验证状态；
- 长期记忆与成长报告；
- 公开资料搜索轨迹和来源状态；
- 静态 Demo 与本地实操版。

### 下一阶段优先级

1. 让 `jd_profile` 成为所有准备、搜索、诊断和成长模块的共享输入；
2. 继续提升证据覆盖率和评分锚点的一致性；
3. 完善训练动作与下一场验证的展示和可追溯性；
4. 增加用户对验证场和证据的人工修正；
5. 在真实样本积累后再评估更复杂的 Agent 工具调用；
6. 最后再考虑多用户、权限和更复杂的岗位相似度能力。

## 10. 一句话总结

Autumn PM Coach 的核心不是“让 AI 给面试打分”，而是：

> **把 JD 要求转化成准备问题，把面试回答转化成证据诊断，把能力缺口转化成训练动作，再通过下一场面试和长期记忆验证是否真正改善。**
