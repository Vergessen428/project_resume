# 评测标准

## V3.1 训练验证边界

训练验证只回答“后续兼容面试中是否出现了可观察改善”，不表示录用概率，也不会修改原场 PM 评分。

验证由后端确定性重建，模型不能直接写入 `action.validation`。验证场必须日期更晚、已完成复盘、岗位族相同，并且公司或归一化岗位名称相同。验证场还必须实际覆盖对应能力；仅仅没有再次出现某个 gap，不能直接判定改善。

- `supported`：能力被实际覆盖、目标 gap 未重复、exact score 提升至少 0.5；
- `partially`：能力被实际覆盖，但 gap 仍存在或分数提升不足；
- `pending`：没有兼容后续面试，或验证场没有足够原文证据。

验证 evidence 只能来自验证场已经通过归一化的技能或子维度证据，不能来自搜索资料、历史面试、模型自述或 outcome。

这份标准把“搜索相关性”“资料可信度”和“面试表现评分”拆成三个不同问题，避免一个分数承担不该承担的判断。

## 1. 搜索相关性

相关性回答的是：**这个搜索结果是否值得候选人打开，来解决当前的信息需求？**

信息需求由公司、岗位、轮次和主题组成。搜索摘要初筛使用固定权重：

| 维度 | 权重 | 判断问题 |
| --- | ---: | --- |
| 公司匹配 | 30% | 是否与目标公司或明确的公司上下文相关？ |
| 岗位匹配 | 25% | 是否与目标岗位/职能相关？ |
| 轮次匹配 | 15% | 是否与目标轮次或面试阶段相关？ |
| 主题匹配 | 15% | 是否涉及当前要查的 case、指标、项目等主题？ |
| 面试具体性 | 10% | 是否像真实面试经历，而不是泛泛求职建议？ |
| 时效性 | 5% | 日期是否仍有参考价值？ |

最终相关度是各项 0–100 分按权重加权后的 0–100 分。它只决定“是否值得人工打开”，不代表原帖真实、准确或可引用。

后端只接受六个子维度并重新计算聚合分；模型或客户端单独提交的 `relevance` 不会作为排序依据。缺少分解时，候选保守降为不可用初筛结果。

系统会优先尝试自动打开限定平台的公开 HTML，并在获得至少 80 个字符可见正文后进入 AI 预审；自动读取正文仍标记为 `auto_fetched_unverified`，不等于人工确认真实性。`auto_approved` 只允许生成问题线索，不能作为事实型复盘上下文；只有人工确认后的 `approved` 才有 `citation_allowed=true`，并记录确认模式、确认时间和当时的正文长度。编辑正文、清除正文或重新 AI 预审都会清除旧的人工确认 provenance。页面脚本壳、访问拒绝或正文不足时，才要求人工打开原帖并补充摘录。`confidence` 是可信度量纲，`relevance` 是相关性量纲，二者不能互换。

这个设计沿用信息检索的基本原则：相关性要针对用户的信息需求，而不是单纯看查询词是否出现在文档里；离线评测还需要查询集合、文档集合和人工 relevance judgments。参见 [Stanford Introduction to Information Retrieval](https://nlp.stanford.edu/IR-book/html/htmledition/information-retrieval-system-evaluation-1.html) 和 [NIST TREC relevance judgments](https://trec.nist.gov/data/reljudge_eng.html)。

`tests/fixtures/relevance_cases.json` 保存脱敏的查询上下文、候选排序和人工相关等级（`0=不相关，1=相关但不足以优先打开，2=高度相关`）。`evaluate_relevance_fixtures()` 只回放后端固定权重，不调用搜索或模型，并输出 Precision@K、Recall@K 和 NDCG@K。它评测“排序是否把值得打开的候选排在前面”，不评测原帖真实性、正文准确性或来源可信度。

当前 `relevance-judgments-2026-07-14-v3` 包含 6 个脱敏场景，离线回放基线为宏平均 Precision@K `0.778`、Recall@K `1.0`、NDCG@K `1.0`。这是可重复的版本化回归基线，不是线上搜索质量、召回率或真实性保证；新增人工判断或调整权重时必须升级数据集版本并重新记录指标。

## 2. PM 面试评分

评分对象是一次面试记录中的**可观察行为**，不是候选人的人格或最终录用概率。项目使用六个维度：

- 产品判断：用户问题、目标、方案取舍和优先级
- 项目主导力：个人职责、关键动作、结果和复盘
- 指标与实验：指标口径、拆解、归因、验证和量化结果
- 推进与协作：分歧、资源约束、风险取舍和闭环
- 结构化表达：结论先行、层次、追问应对和简洁度
- 业务与岗位理解：个人经历与 JD、公司业务、岗位场景的连接

每个维度使用 1–5 行为锚点：

| 分数 | 标准 |
| ---: | --- |
| 1 | 没有展示相关行为，或回答与问题明显不匹配 |
| 2 | 提到了相关点，但缺少关键定义、个人动作、取舍或结果 |
| 3 | 基本合格，有例子和判断，但仍有一处重要环节不完整 |
| 4 | 行为具体，能解释取舍、协作和结果，仅有少量细节缺口 |
| 5 | 证据闭环，能连接用户/业务、决策、指标、结果和复盘 |

每个结论必须先在转写中找到原文证据。证据不存在时，结论应省略或标记“无转写原文可佐证”；记录太短时减少输出项，不用默认 3 分填满报告。

这套维度与公开 PM 招聘材料中常见的 product thinking、problem solving、metrics/data、collaboration/communication、execution/ownership 等能力重叠，但分数只用于个人复盘和练习优先级，不作为招聘决策。参考 [Product-Led Alliance 的 PM scorecard](https://www.productledalliance.com/content/files/2026/01/Hiring-a-product-manager_-Interview-questions.pdf)。

## 3. 可复算的质量指标

- 搜索候选：记录每个候选的相关度分解、初筛理由和真实 URL。
- 资料库：记录原文摘录、可信度、限制项和人工确认状态。
- 单场复盘：记录每条结论的证据、评分锚点和下一步练习。
- 跨场趋势：只统计已经完成复盘的记录；样本不足时显示数据不足，不生成确定性结论。

## 4. 子维度和证据覆盖度

六项 PM 能力各包含四个子维度，固定权重见 `skills/pm-interview-evaluation/references/pm-rubric.md`。后端始终输出六项能力；没有覆盖的能力或子维度显式标记为 `missing`，不能把缺失静默当成低分。对 V2 输出，`exact_score` 只由有合法状态和可核对评分的 `observed / contradicted / unverified` 子维度按固定权重计算；`missing` 与 `not_applicable` 永远不贡献分数，即使模型返回了分数字段也会被丢弃。非法 ID 被丢弃，维度分数被限制在 1–5。新模型输出缺少子维度时不会退回使用模型随手给的技能总分；只有旧格式兼容读取才保留历史技能分数。

单项 `evidence_coverage` 是有可核对证据子维度权重 / 适用子维度权重；整场覆盖度是六项单项覆盖度的平均值。覆盖度达到 0.75 且至少有两条有效证据时，单项诊断才可达到高置信度；没有转写证据时必须是低置信度。`coach_score` 只平均本场实际可计算的 `exact_score`，缺失能力不会被伪装成 1 分；它是训练信号，不是招聘评分。

## 5. 训练验收

每个重点缺口需要给出一个练习动作、一道重写题、最多四条可观察成功标准和一条追问。行动项保存 `pending / passed / needs_retry`、完成时间和备注，并可按 `pre_test → rewrite → post_test` 保存最多 12 次训练记录。后端强制阶段顺序；没有 `post_test` 时，API 和前端都拒绝将行动标记为 `passed`，不能把它解释为完整训练闭环。通过验收的含义是下一次回答满足这些标准，而不是模型分数自动提高。跨场聚合只使用已完成复盘中的规范化 `canonical_gap_id`，开放行动会保留验收标准和下一次验证问题。

训练记录只保存用户主动提交的回答、可选自评和验收备注，不自动把模型判断写成客观成绩。后续可以在不改变事实证据的前提下增加模型解释，但模型不能跳过阶段门槛或直接修改行动状态。

## 6. 评分可追溯性与结果反馈

训练行动项会保留来源面试 ID、日期、公司和轮次；长期报告展示训练建议的来源，支持回溯“诊断 → 训练 → 下一场验证”。这条来源链只说明建议从哪场复盘产生，不把训练完成或自报结果解释为招聘真值。

每次复盘保存 `scored_by`，包括实际成功调用的 provider、model、prompt 版本、rubric 版本和时间。历史没有这些字段时标记为 `legacy_unknown`，不补猜模型，也不修改历史分数。跨场趋势遇到模型、prompt 或 rubric 混用时必须标记不可直接比较；阶段报告的解释层也由后端强制追加“趋势仅供参考”的边界提示，不能只依赖模型自行说明。

前端可选开启发送前脱敏：后端只对用户填写的公司名做字符串替换，并将替换后的副本交给复盘模型，原始本地记录不被覆盖。该开关是尽力而为的降风险措施，不等同于完整匿名化。

面试结果只保存用户主动填写的 `passed`、`failed` 或 `pending`，并要求 `outcome_source=self_reported` 才能进入聚合；旧记录缺少来源字段时不会被默认当作自报结果。4 场有明确结果时才展示描述性分组，6 场且通过/未通过各至少 2 场时才展示较稳定的方向性观察。所有结果都要显示样本数和限制，不能解释为评分造成了通过或未通过，更不能当作录用预测。

## 7. 数据处理边界

本地保存与外部处理是两件事：JD、简历、转写和个人笔记默认写入本机 JSON；用户主动点击复盘、搜索或转写后，必要字段会发送到配置的模型或搜索服务。静态 Demo 不发送真实数据。项目不把“替换公司名”称为完整脱敏，后续 PII 提示也只能作为 best-effort 风险提醒。

前端的“可能包含敏感信息提示”只覆盖手机号、邮箱、身份证号等有限模式，属于发送前的非阻塞提醒；它不自动替换、不拦截请求，也不保证发现所有个人信息。真实使用前仍需由用户删减内容并确认外部服务的处理边界。

本地数据生命周期包含版本化 JSON 导出、备份、恢复和选择性清理。当前备份带 SHA-256 完整性摘要，旧版备份会明确标记为未验证；恢复前先校验版本、标识、结构和摘要，跨 store 写入失败时尝试回滚。清除转写或面经正文会保留结构化诊断、来源元数据和清理标记，清理后仍可编辑 outcome 和其他元数据；备份恢复需要显式确认，备份文件默认位于本地 `app/data/backups/` 并被 Git 忽略。音频和简历文件只进入受控临时目录，处理完成后删除，启动时清理已知前缀的异常残留。保留策略通过 `/api/data/retention` 先预览、再显式确认执行，默认不自动运行，且只允许清理原始转写与面经正文。

## 8. 固定评测与回放

`tests/fixtures/evaluation_cases.json` 覆盖稀疏转写、伪造引用、非法能力/缺口 ID、矛盾指标和旧 schema。`tests/fixtures/research_cases.json` 会回放空搜索、脚本壳、自动读取正文但未验证、重定向越界和网页诱导样例。`app/core/evaluation_harness.py` 提供四类单独入口和 `evaluate_all_fixtures()` 统一入口，可输出可机读回归报告；它们评测的是安全契约，不把少量 fixture 伪装成人工标注的模型质量结论。

研究 Agent 的回放不联网、不调用模型，由 fixture 注入搜索与读取结果，但仍执行真实的查询预算、白名单、规范 URL 去重、来源状态和空结果解释。运行 `evaluate_research_fixtures()` 可以验证候选不会因为搜索摘要或网页文本直接进入可信资料状态。

`tests/fixtures/quality_gold_cases.json` 是最小人工标注约束集：当前 5 个匿名场景覆盖稀疏指标归因、完整实验、多轮量化缺失、被追问后推翻的因果结论，以及 JD 业务适配缺失。它为关键能力给出可接受分数范围、必需/禁止缺口、最少可核对证据、量化缺失、多轮追问中的已验证证据和训练行动可执行条件，并保存不同模型输出及应被拒绝的坏输出。`evaluate_quality_gold_fixtures()` 会报告约束是否满足、坏输出是否被识别、以及可比较模型之间的分数漂移。标注版本、样例和阈值必须由人维护；这不是招聘真值，也不能替代人工复核。

研究 Agent 对公开资料还会提取有限的 `question_leads`。这些线索必须带有来自原文的可核对摘录，否则标记为 `unverified`；它们只进入 JD/面后速记的出题上下文，不得进入本场面试的事实证据或评分。保存面试时会保留受限的研究快照（相关度、来源状态、线索主题和查询），但不会把公开原帖正文复制进面试记录。

速记问题接口同时返回 `research_meta`，说明本次实际提供了多少条研究上下文、其中多少条通过人工确认门槛；没有研究上下文时明确标记为 `jd_resume_fallback`，页面不得暗示已经完成公开搜索。
快照会保留研究记录 ID；每次出题或复盘前重新对照资料库当前状态，已删除或降级的来源只能保留为历史线索，不能继续作为 `approved_context`。

长期记忆由确定性聚合器生成 `memory_version`、`generated_at` 和 `audit`。`audit` 只记录输入面试 ID、schema/评分身份、治理覆盖和算法版本，不包含转写正文；`/api/growth-memory/replay` 可回放这次聚合的输入边界。

阶段成长报告必须以这份确定性长期记忆为结构化事实源：重复缺口、出现次数、能力趋势、来源面试和优先训练行动由后端投影，模型最多补充与已存在对象匹配的解释文字。报告通过 `report_grounding` 标注记忆版本、算法版本、输入场次数和结构化字段数量；无法在记忆中找到的模型自报 pattern、次数或行动会被丢弃。这样“模型解释”与“系统统计”可以分别审计，不会把一段生成文本伪装成跨场证据。
