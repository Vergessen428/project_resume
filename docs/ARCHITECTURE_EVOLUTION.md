# 架构演进说明：从单机 JSON 到可规模化

> 这份文档回答面试官常问的一句话：「你现在用本地 JSON 存数据，如果要上线给很多人用，
> 你会怎么改？」核心不是「JSON 很low要换掉」，而是**为什么现在这样够用，以及边界在哪、
> 怎么演进**。

## 现状：为什么是本地 JSON

三个 store（[interview_store.py](../app/core/interview_store.py)、resume、research）都是
「单文件 JSON + 线程锁 + 原子写」：

```python
def _write_all(self, records):
    tmp = self.path + ".tmp"
    with open(tmp, "w", encoding="utf-8") as f:
        json.dump(records, f, ensure_ascii=False, indent=2)
    os.replace(tmp, self.path)   # 原子替换，避免写到一半崩溃损坏文件
```

这是**刻意的 MVP 取舍**，理由成立：

1. **单用户场景**：这是「我自己的求职助手」，同一时刻只有一个人用，没有并发争用。
2. **数据量小**：一个人一季秋招几十场面试，JSON 全量读写的开销可以忽略。
3. **零依赖、零运维**：纯标准库，不用装数据库、不用管连接池和迁移脚本，`git clone` 就能跑。
4. **可移植、可检查**：数据就是纯文本，肉眼可读，备份就是拷文件。

在需求没被验证之前就上数据库，是典型的过度设计。

## 现状的瓶颈（什么时候必须换）

诚实地讲清楚边界，比假装没有问题更能加分：

| 瓶颈 | 触发场景 | 现在为什么不痛 |
|---|---|---|
| **全量读写** | 每次改动都重写整个 JSON，O(n) | 单人数据量小，n 很小 |
| **并发写** | 多请求同时写同一文件 | 单用户，几乎不并发；且有锁兜底 |
| **多实例** | 水平扩容成多个进程/机器 | 单机单进程，`render.yaml` 也限制单实例 |
| **查询能力** | 只能全量加载后在内存过滤，无索引 | 数据小，内存过滤够快 |
| **多租户** | 多个用户各自的数据隔离 | 产品定位就是单人 |

一句话：**只要还是「单人、单机、小数据」，JSON 就是最优解；一旦要「多人 / 多实例 /
大数据 / 复杂查询」，就必须换。**

## 演进路径（按成本递增）

### 第一步：SQLite（最小改动，解决持久化与查询）

- 换掉三个 store 的底层实现，接口（`create/get/list/update`）保持不变，上层 web_app 不用动。
- 收益：真正的事务、索引、按条件查询（按公司/状态/日期筛选面试），单文件仍然零运维。
- 适用：数据量涨到几千条、需要筛选/排序、仍是单机部署。
- 成本：低。SQLite 是 Python 标准库自带（`sqlite3`），无新依赖。

### 第二步：Postgres + 连接池（支持多实例与并发）

- 触发点：要水平扩容（多个 Render 实例）、或引入多用户。
- 需要：抽一层 `Repository` 接口，SQLite/Postgres 各一个实现；引入连接池；写 schema 迁移。
- 同时要做的配套：把「单用户单一 workspace」升级成「用户表 + 数据按 user_id 隔离」，
  鉴权从单口令换成正式的账号体系（OAuth / 邮箱登录）。

### 第三步：对象存储 + 队列（重负载 AI 任务）

- 触发点：AI 复盘/转写量大、耗时长，同步请求会拖垮 Web 进程。
- 演进：上传文件放对象存储（S3 等），AI 任务丢进队列（Celery/RQ）异步处理，前端轮询/推送结果。

## 面试怎么讲（30 秒版）

> 我现在用本地 JSON + 原子写 + 锁，因为这是单人单机的 MVP，数据量小，上数据库是过度设计。
> 但我很清楚它的边界：全量读写、无法多实例、没有索引查询。演进路径我也想好了——先换 SQLite
> 拿到事务和查询能力且接口不变，再到 Postgres 加连接池和多用户隔离、把单口令换成账号体系，
> 最后 AI 任务异步化。关键是每一步都由真实需求触发，而不是为了技术而技术。
